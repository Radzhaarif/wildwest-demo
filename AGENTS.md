# AGENTS.md

Правила для помощника, который работает с этим проектом. Цель файла - сохранить проект понятным, не ломать уже работающие сценарии и делать изменения маленькими, проверяемыми шагами.

## Role

Ты помогаешь с разработкой проекта. Работай осторожно, маленькими проверяемыми изменениями.

Перед тем как менять файлы, сначала пойми текущую механику: прочитай релевантные документы, затем код и данные, которые затрагивает задача.

## Project Rules

- Перед изменениями изучи релевантные файлы.
- Не переписывай архитектуру без явного запроса.
- Не добавляй новые зависимости без отдельного обоснования.
- Не меняй публичные API, формат JSON/JSONC-данных или DOM-контракт без предупреждения.
- Не удаляй существующий функционал ради упрощения.
- Если задача неоднозначна, сначала перечисли допущения.
- Если видишь старые папки `back_*`, считай их резервными снимками, а активной версией - файлы в корне проекта.
- Не меняй ассеты, локали и балансные значения "заодно", если задача про другое.

## Project Context

- Проект запускается как статический браузерный сайт.
- Основной код игры находится в `src/map-module.js`.
- Основной HTML-контракт находится в `index.html`.
- Основные стили находятся в `src/styles.css`.
- Настраиваемые данные лежат в `data`.
- Документ с устройством проекта: `data/README.md`.
- Журнал изменений: `VERSIONS.md`.
- Архитектурные решения: `DECISIONS.md`.

## Code Style

- Сохраняй существующий стиль проекта.
- Минимизируй diff.
- Имена переменных должны объяснять смысл, а не настроение автора в 3 ночи.
- Комментарии добавляй только там, где они объясняют неочевидную причину, а не пересказывают код.
- Не смешивай рефакторинг и новую функцию в одной правке без причины.
- Для ручных правок используй точечные изменения, не форматируй большие файлы целиком.

## Data And Assets

- JSONC-файлы поддерживают комментарии отдельными строками через `//`.
- Обычные JSON-файлы должны оставаться валидным JSON без комментариев.
- В карте стандартный путь к ассетам - прямой путь от `data/`, например `data/Assets/events/trader.png`.
- Если добавляешь новый ключ локали, проверь все активные локали из настроек.
- Если меняешь структуру данных, обнови README и добавь контекст в `VERSIONS.md`.

## Testing

- После изменения логики запускай доступные проверки.
- Если автоматических тестов нет, предложи минимальную ручную проверку в браузере.
- Для изменений JSON/JSONC проверь, что файлы парсятся.
- После правок русскоязычной документации или JSONC-комментариев запускай `node scripts/check-encoding.mjs`, чтобы поймать сломанную кодировку до ответа пользователю.
- Для изменений UI проверь основной сценарий вручную или через браузер, если это возможно.

## Documentation

- Если меняется поведение системы, обнови `data/README.md`.
- Если меняется формат данных, запуск, структура файлов или пользовательский сценарий, обнови `data/README.md`.
- Каждое значимое изменение фиксируй в `VERSIONS.md`.
- В записи версии добавляй не только список изменений, но и контекст версии: зачем изменение сделано и почему его не стоит менять без причины.
- Если принято архитектурное решение, добавь запись в `DECISIONS.md`.

## Final Response

В конце работы укажи:

- что изменено;
- какие файлы затронуты;
- какие проверки выполнены;
- какие риски остались.
