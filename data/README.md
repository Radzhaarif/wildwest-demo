# Контекст системы данных

Этот документ фиксирует текущее устройство прототипа, чтобы при дальнейшей работе быстро восстановить контекст. Основной код игры находится в `src/map-module.js`, а настраиваемые данные лежат в папке `data`.

## Важное правило сопровождения проекта

Правила работы с проектом для помощника находятся в `AGENTS.md`. Перед значимыми изменениями нужно учитывать эти правила.

Все значимые изменения кода, JSON/JSONC-данных, ассетов, интерфейса и игровых функций нужно фиксировать в `VERSIONS.md`.

Если изменение добавляет новую функцию, меняет существующую механику, правила данных, запуск проекта, структуру файлов или пользовательский сценарий, этот README тоже нужно обновить. README должен оставаться актуальной картой проекта, а `VERSIONS.md` - журналом того, что и когда поменялось.

В `VERSIONS.md` нужно писать контекст версии: не только что добавлено, но и зачем это добавлено, какую проблему решает и почему это не стоит менять без отдельной причины.

Если принимается архитектурное решение, его нужно записать в `DECISIONS.md`.

Важное направление на будущее: проект сейчас остается статическим web-прототипом, но его нужно развивать так, чтобы позже можно было упаковать игру в desktop `.exe` и потенциально выложить в Steam. Поэтому не стоит без отдельного решения завязывать игру на внешний сервер, CDN, dev-only запуск или API, которые плохо работают в packaged-билде. Ассеты и данные должны оставаться локальными, а систему сохранений и пути к файлам нужно проектировать с учетом будущей desktop-оболочки.

## Общая идея

Игра строится как рогалик с путешествием по карте между точками. Игрок выбирает доступную точку, получает событие, после события возвращается на карту и может идти только по соединенным путям.

Основные модули:

- Карта: экран карты, доступные переходы, события и возврат из боя.
- Генерация карты: отдельный модуль `src/map/map-generation.js`, который строит уровни, точки, payload событий и дороги.
- Бой: отдельный match-3 модуль. Карта передает ему `BattleRequest`, получает `BattleResult` и сама решает прогресс маршрута.
- Магазин: событие торговца с покупкой предметов за золото.
- Лечение и награды: события, которые меняют состояние игрока.

Сейчас главный рабочий модуль - карта с событиями `skip`, `shop`, `heal`, `reward` и боевыми вариантами `battle`, включая boss-события как именованные battle-точки.

## Как загружается игра

Корневой `version.json` содержит только поле `version`. Это общий cache-version для CSS, JS-модулей, JSON/JSONC-данных, музыки и локальных ассетов. Если меняется код, данные или ассет, версию нужно поднять вместе с изменением; если версия не менялась, браузер имеет право использовать прежние cached-файлы.

Страница `index.html` сначала читает `version.json` с `cache: "no-store"`, затем подключает `src/styles.css`, `src/map-module.js` и import map для остальных JS-модулей с параметром `?v=<version>`.

При загрузке:

1. Игра читает `default-settings.json`.
2. Игра читает `current-settings.json`.
3. Если в браузере есть сохраненные настройки в `localStorage`, они перекрывают пользовательские настройки из файла. Структурные поля `languages` и `rewardAnimationMs` всегда берутся из JSON-файлов, чтобы их можно было менять без очистки `localStorage`.
4. Игра читает `data/settings/load.jsonc` и применяет оформление loading-screen. Если файл не загрузился, используется встроенный fallback.
5. Загружается локаль `data/locales/<language>.json`.
6. Читаются `data/settings/campaign.jsonc`, `data/settings/items.jsonc`, `data/player/experience-table.jsonc` и `data/player/cheats.json`.
7. До показа главного меню запускается проверка данных: кампания, каталог предметов, таблица опыта, конфиг чит-команд, карты из кампании, включенная карта `campaign.tutorial`, включенная `cheats.testRun` карта, враги из активных battle-точек, `data/settings/battle-ui.jsonc`, стартовое состояние игрока, test-run состояние игрока и все локали из `settings.languages`.
8. Если проверка прошла, предзагружаются локальные ассеты активной кампании, обучения и включенной test-run карты: картинки карт, иконки событий, `items.jsonc` icon/bigIcon/sound_effect, `map-ui`, `battle-ui`, фоны боев, изображения врагов, награды, звуки, звук активации читов, ассеты меню, логотип загрузки и другие пути из `data/Assets`. Ассеты скачиваются в общий RAM-кэш как `blob:` URL, а изображения дополнительно декодируются и удерживаются в памяти для быстрого повторного использования.
9. До показа главного меню заранее загружается код боевого модуля: `battle-module.js`, `battle-engine.js`, `battle-view.js` и зависимости. Сам бой при этом не создается заранее.
10. Если часть картинок или звуков не загрузилась, игра пишет список проблем в консоль и продолжает запуск. JSON-ошибки остаются фатальными через валидатор.
11. Показывается главное меню.

При нажатии `START`:

1. Игра использует уже загруженные и проверенные данные. Если по какой-то причине данные не готовы, START выполняет аварийную догрузку тем же путем, что и boot.
2. `default-player-state.json` копируется в состояние игры в памяти.
3. Для забега создается 16-символьный `run` seed. Если страница открыта с параметром `?seed=<16 символов>` или `?runSeed=<16 символов>`, используется он; иначе seed генерируется заново.
4. В глобальный лог добавляется новый разделитель забега с номером забега, `run` seed и результатом проверки данных.
5. Загружается стартовая карта из `campaign.startMapId`.
6. Для карты создается производный seed с именем `map:<mapId>:<campaignIndex>`, он пишется в лог и передается в генератор карты.
7. Генерируется карта и запускается вступительная прокрутка сверху вниз.

Если в главном меню ввести чит-пароль `iddqd`, появляется скрытая кнопка smoke test. Она запускает отдельный test-run из `data/player/cheats.json`, загружает `data/player/smoke-test-player-state.json` и карту `data/maps/SmokeTest.jsonc`. Обычный `START`, `campaign.maps`, счетчик карт кампании и прохождение основной кампании от этого не меняются.

Кнопка `ОБУЧЕНИЕ` всегда стоит между обычным стартом и настройками. Она запускает `campaign.tutorial` как отдельную карту `1/1`, не меняя маршрут и счетчик основной кампании. Учебный run начинается с нулевыми `gold`, `item_skull`, `item_swap` и `item_time`; обычный стартовый инвентарь кампании не затрагивается. Карта образует линейный маршрут из двенадцати точек: вводный разговор с Капитаном, два урока комбинаций с Долли, сундук, учебный торговец, доктор, разговор об активных предметах, третий бой с препятствиями, повышение уровня, разговор о ярости, финальный бой с Капитаном и его напутствие. Все учебные Долли используют `Dolly.png`, имеют независимый учебный баланс и дают по 20 опыта: первая имеет `100` HP и наносит `1` урон, вторая — `200` HP и `2` урона, третья — `300` HP и `2` урона. После торговца третий бой сохраняет 5 черепов, 5 перчаток и 80 золота, нужные для урока. Пока первая точка не выбрана, над ней пульсирует желтая светящаяся стрелка. После финального окна завершения кнопка `OK` возвращает игрока в главное меню.

Бой `tutorial_battle` не использует обычный стартовый сценарий Долли. Он начинается с `80` HP игрока, `90` единиц лечения, `60/100` агрессии и `100` HP у `tutorial_Dolly`; удар при заполнении агрессии наносит `1` урон. Пять шагов из `battle[].tutorial.steps` последовательно заменяют поле: обязательный swap ножей, бинтов, мусора и щитов, затем обязательное ручное перемешивание. На swap-шаге доступны только `from`, `to` и `matchCells`; остальные клетки затемнены и не принимают ввод. Управляемый swap разрешает ровно одну волну совпадения: после её штатных анимаций возможные случайные каскады из refill не выполняются, потому что поле сразу заменяется следующим учебным шагом. После перемешивания подсказка снимается, а бой продолжается как обычный свободный match-3 с сохранёнными HP, лечением, агрессией и здоровьем Долли.

После победы и получения награды Капитан открывает второй урок. `tutorial_advanced_battle` запускает отдельную Долли с `200` HP и уроном `2`, затем последовательно обучает линии из четырёх пуль, квадрату `2x2`, совместной комбинации обычных и улучшенной пули, линии из пяти пуль и применению генератора к бинтам. Комбинации из четырёх создают `bullet_power`, комбинация из пяти — `battary`. На шаге `battery` игрок выбирает генератор и соседний бинт; генератор активирует все бинты на поле с подавленной агрессией. Учебное поле содержит `36` бинтов: при стартовых `90` HP и `20/100` лечения их суммарные `180` единиц лечения дважды заполняют шкалу и доводят игрока до `100` HP, не меняя `80/100` агрессии. Затем обязательный shuffle наносит `2` урона и открывает свободный бой против оставшихся `115` HP Долли.

Третий бой запускает Долли с `300` HP и уроном `2` и обучает препятствиям штатными действиями поля: две неудачные попытки поменять бинт и гранату через стену, перчатка (`freeSwap`) для того же обмена, две двухкликовые попытки провести нож из лозы к бинту по направленной стрелке, вторая перчатка для переноса дальнего ножа на место бинта и вертикальной тройки через лозу, три обычных хода под уроном четырёх горящих бочек (`flame`), череп (`skull`) по центру их области `3x3` и два учебных применения золота (`gold`). На шагах активных предметов указатель сначала стоит на обязательном слоте инвентаря, затем ведёт к первой и второй клетке: у второй перчатки это дальний нож и бинт, у черепа — пуля. Сначала щит между двумя бинтами гарантированно превращается в бинт, затем бочка между двумя щитами — в щит. Обе созданные тройки проигрывают полную первую волну штатных match-анимаций, эффектов, удаления и добора до перехода дальше; случайный последующий каскад не запускается. Предмет в лозе снимается с покрытия, когда его клетка входит в обычное совпадение; после обязательного shuffle бой, как и предыдущие уроки, становится свободным.

Три награды по 20 опыта дают суммарные 60 опыта и впервые пересекают порог уровня 50 только после третьей Долли. У её battle-варианта необязательный `levelUpTutorial` заменяет обычные три случайных варианта четырьмя фиксированными: `item_bullet`, `item_Shield_power`, `red` и `green`. Эти четыре карточки показываются квадратом `2x2`. Сначала окно показывает общий текст `introTextKey`; затем игрок должен нажать каждую карточку и прочитать её `textKey`. Только после осмотра всех четырёх карточек разрешается выбрать одну награду и забрать её; повторный клик по любой карточке всегда снова показывает её пояснение, в том числе после разблокировки выбора. Обычные level-up события без `levelUpTutorial` сохраняют случайный выбор по таблице опыта. После выбора на карте открывается разговор Капитана о таймере ярости и особом приёме врага.

Финальный `tutorial_captain_battle` использует две отдельные конфигурации одного персонажа. Первая попытка запускает `tutorial_Captain`: 200 HP, 5 стен, 4 лозы, 4 коробки, 5% шанс заменить новый обычный предмет горящей бочкой и ярость через 10 секунд с фиксированным уроном 1000. После неизбежного поражения учебная кнопка переигровки подменяет только этот бой на `tutorial_Captain_retry`: 100 HP, те же количества препятствий и шанс бочек, 10 щитов, ярость через 60 секунд с уроном 10. Повторная попытка выдаёт 5 часов и проводит через три шага: boardless `clock`, детерминированную тройку ножей против щитов и обязательный `shuffle`. Три ножа снимают три единицы щита, потому что каждый щит поглощает одну атакующую фишку целиком. Учебные поля щитов и shuffle сохраняют 5 стен, 4 коробки и 4 лозы; после перемешивания остаётся свободный бой. Повторные рестарты остаются на облегчённой конфигурации; у обычных боёв подмены врага нет.

Необязательный объект `battle[].tutorial` содержит `enabled`, ключи `titleTextKey`/`wrongMoveTextKey`, `teacherImage`, необязательный объект `playerInventoryQuantities` и непустой массив `steps`. Пока учебный бой активен, длительное наведение на предмет поля показывает микроподсказку с его текущей эффективностью: урон/лечение/изменение агрессии учитывают инвентарь игрока и модификаторы текущей стадии врага, генератор описывается как массовая активация, а бочка сообщает урон за каждый обычный ход. В обычных боях такие подсказки поля не подключаются. `playerInventoryQuantities` переопределяет количество указанных предметов при входе в учебную попытку, но не восстанавливает потраченные предметы при переходах между шагами; в первых двух боях Tutorial он обнуляет `gold`, `item_skull`, `item_swap` и `item_time`. Третий бой не задаёт override и использует реальные ресурсы, купленные на предыдущей точке, поэтому перчатка, череп и золото тратятся штатно. Каждый шаг задаёт уникальные `id`, `action`, `textKey`; для всех действий, кроме `clock`, нужна прямоугольная матрица `board`. Поддерживаются `swap`, `battery`, `shuffle`, `blockedWall`, `blockedVine`, `freeSwap`, `skull`, `gold` и `clock`. Для `swap`/`freeSwap` обязательны `from`, `to` и минимум три `matchCells`; `battery` требует координаты генератора/цели и минимум две подсвеченные клетки; `blockedWall` требует `from`, `to`, массив `walls` и `requiredAttempts`; `blockedVine` требует `from`, массив `vines` и `requiredAttempts`, а необязательный `to` переводит попытку в двухкликовый обмен и рисует направленную стрелку; `freeSwap`/`skull`/`gold`/`clock` требуют `requiredItemId`, а череп и золото — координату `to`. Необязательный `replacementItemId` у шага `gold` задаёт фиксированный результат только этой учебной замены; без него обычный и учебный gold используют стандартный случайный пул. Если gold создаёт совпадение, учебный flow разрешает полностью отыграть одну волну каскада и блокирует только случайные последующие волны. Шаг `clock` сохраняет случайно сгенерированное поле и препятствия, блокирует их до клика по требуемым часам и затем либо открывает следующий шаг, либо полностью перерисовывает разблокированное поле. Для детерминированного шага можно явно задать `walls`, `boxes` и `vines`; это заменяет случайные препятствия на указанные координаты. Необязательный `guideCells` оставляет дополнительные учебные объекты яркими и подсвеченными, но не превращает их в разрешённые действия. Значения предметов в `board` и `replacementItemId` должны быть существующими `itemId`. Поля `playerHealthCurrent`, `playerHealCurrent`, `enemyHealthCurrent` и `enemyAggressionCurrent` необязательно переопределяют состояние при входе в конкретный шаг. `wrongTextKey` шага может заменить общую подсказку о неверном действии.

У battle-варианта необязательный `tutorialRetry` включает учебную подмену после первого поражения: `enabled: true`, новый `enemyId`, `defeatTextKey` и обычный вложенный `tutorial`. Оба enemy-файла предзагружаются до открытия боя. Подмена выполняется один раз внутри текущего battle overlay и не меняет поведение стандартной кнопки переигровки без этого payload.

У shop-варианта необязательный объект `tutorial` включает обязательный сценарий покупки: `enabled: true`, `requiredItemIds` с непустым списком обязательных товаров и `wrongTextKey`. Товарные карточки, торговец, инвентарь и отказ от подтверждения остаются обычными; затемнены и заблокированы только выход и кнопка покупки, причём покупка включается после выбора всех обязательных предложений. У каждого ещё не выбранного обязательного предложения жёлтая светящаяся стрелка указывает прямо на кнопку `Выбрать`; после выбора всех обязательных предложений указатель переходит на `Купить`. В Tutorial это два набора по 5 предметов: `item_skull` и `item_swap`, каждый за 10 золота. У heal-варианта `healPercent: 100` лечит на полный максимальный запас здоровья — при текущем игроке это 100 HP.

Важно: браузер не записывает изменения обратно в JSON-файлы. Текущее состояние игрока и выбранные настройки живут в памяти браузера и `localStorage`.

Лог событий намеренно глобальный для текущей сессии браузера: при новом START он не очищается, а получает новый разделитель забега. Это помогает сравнивать несколько попыток подряд.

## Проверка проекта

Перед значимыми изменениями и перед ручной проверкой в браузере можно запускать общий чекер:

```bash
node scripts/check-project.mjs
```

Скрипт выполняет набор быстрых локальных проверок без запуска браузера: `node --check` для JS/MJS-файлов из `src` и `scripts`, парсинг всех `data/**/*.json` и `data/**/*.jsonc`, `validateGameData()` через локальный `fetch`, проверку существования ссылок на `data/Assets`, равенство ключей активных локалей, баланс CSS-скобок, `scripts/check-battle-engine.mjs` и `scripts/check-encoding.mjs`.

Для основного UI-сценария есть отдельный browser-smoke:

```bash
node scripts/browser-smoke.mjs
```

Для полной проверки одним вызовом можно включить smoke-флаг:

```bash
node scripts/check-project.mjs --with-smoke
```

Без флага `check-project` не запускает браузер намеренно: он остается быстрым структурным чекером. С `--with-smoke` он после структурных проверок запускает `scripts/browser-smoke.mjs --start=smoke-test`. Сам полный smoke поднимает временный static server, открывает Chrome/Edge через DevTools Protocol, вводит `iddqd`, запускает скрытую SmokeTest-карту и линейно проходит все основные активности карты.

Логотип лежит в `data/Assets/logo.png` и попадает в общий RAM-кэш ассетов.

## Кампания

`data/settings/campaign.jsonc` задает список карт:

- `id` - идентификатор кампании.
- `nameTextKey` - ключ названия кампании в локалях.
- `startMapId` - с какой карты начать.
- `maps` - список карт.
- `tutorial` - необязательная отдельная карта обучения, которая не входит в `maps` и не меняет обычный маршрут кампании.

У `tutorial`:

- `enabled` - показывает или скрывает кнопку обучения и включает карту в общую валидацию/предзагрузку;
- `buttonTextKey` - ключ подписи кнопки главного меню;
- `mapId` и `config` - id и путь к JSONC карты обучения;
- `startingInventoryQuantities` - необязательные количества предметов только для отдельного tutorial-run; в текущем обучении обнуляет золото и активные предметы;
- `onComplete` - отдельный `victory`-текст завершения обучения; необязательный `returnToMainMenu: true` привязывает закрытие этого окна к возврату в главное меню.

У каждой карты в кампании:

- `mapId` - идентификатор карты.
- `config` - путь к JSONC карты.
- `onComplete` - что делать после прохождения карты.

Типы завершения:

- `nextMap` - загрузить новую карту по `nextMapId`.
- `victory` - показать победный текст из локали.

Служебная `SmokeTest` карта не добавляется в `campaign.maps`. Она находится в обычных data-файлах и проходит валидатор/предзагрузку, но запускается только через `data/player/cheats.json.testRun`, чтобы не менять обычный маршрут игрока и счетчик карт кампании.

## Карта

Основная карта описывается в `data/maps/WildWest.jsonc`. Служебная `data/maps/SmokeTest.jsonc` использует тот же контракт карты, но содержит короткий линейный QA-маршрут: `dialog`, `skip`, магазин со всеми безопасными предметами категории `item`, платное лечение, смешанную reward-награду, бой с врагом `test` и финальный boss-бой. Для удобства полного smoke-run магазин SmokeTest продает предметы пачками по 10 за цену одной позиции, а лекарь полностью восстанавливает HP.

Основные поля:

- `id` - идентификатор карты.
- `nameTextKey` - ключ названия карты в локалях.
- `mapImage` - фоновое изображение карты.
- `firstNodeArrow` - если `true`, до первого выбора показывает над первой доступной точкой желтую пульсирующую стрелку-подсказку.
- `pathRules.connectionCountWeights` - общие весовые правила ветвления дорог карты: сколько исходящих путей точка пытается получить чаще или реже.
- `pathRules.maxSinglePathChain` - сколько одиночных переходов подряд допускается, прежде чем генератор начнет пытаться сделать развилку.
- `events` - общий каталог событий карты. У каждой записи есть `name`, `type` и `icon`; весов здесь нет.
- `levels` - массив уровней карты, включая финальные boss-события, если они нужны. Каждый уровень хранит количество точек, правила дорог и ссылки на события из общего каталога.
- `levels[].nodes.count` - фиксированное количество точек на уровне.
- `levels[].nodes.min/max` - случайное количество точек на уровне, если `count` не указан.
- `levels[].paths` - правила генерации дорог от этого уровня к следующему.
- `levels[].events` - ссылки на события уровня. У каждой записи есть `name` из общего каталога, `weight` для случайного выбора и `guaranteed` для обязательной точки.
- `battle[].background` - конкретный задник боя для отдельного battle-варианта. Общий список `battleBackgrounds` больше не является стандартом.
- `battle`, `reward`, `heal`, `shop`, `skip`, `dialog` - payload-варианты конкретных событий. Каждый вариант должен иметь `eventName`, который ссылается на `events[].name` с таким же `type`.
- Финальная точка карты задается обычным событием в `levels[]`; отдельный объект `boss` больше не обязателен.

Пути к ассетам в карте пишутся прямым путем от папки `data`, например:

- `data/Assets/map/WildWest_map.png`
- `data/Assets/backgrounds/WildWest_1.png`
- `data/Assets/enemy/Boss_1.png`
- `data/Assets/enemy/Sweety.png`
- `data/Assets/events/trader.png`
- `data/Assets/item/item_gold.png`

Короткий формат `assets/...` больше не является стандартом для карт. Если в JSON добавить новую папку внутри `data/Assets`, код не нужно менять: достаточно указать полный путь от `data/`.

## Каталог предметов

Все предметы и HUD-ресурсы описаны в `data/settings/items.jsonc`. Это единый источник правды для названий, описаний и иконок предметов: код больше не строит путь к иконке из `itemId`.

Поля предмета:

- `itemId` - идентификатор, который используется в картах, наградах, магазине, инвентаре и `playerState`.
- `category` - свободная категория для навигации по каталогу, например `currency`, `map_item`, `match-3`, `rare_match-3`, `stat`.
- `limitByInventory` - необязательный флаг ограничения предмета по текущему количеству в инвентаре. Если `true`, а у игрока уже есть не меньше предметов, чем задано в `data/player/experience-table.jsonc.limitedItemInventoryThreshold`, предмет не попадает в предложения повышения уровня. В reward-наградах карты/врага и товарах магазина предмет остается видимым, но получает красную плашку MAX/МАКС, затемненную иконку и не может быть куплен или начислен при получении reward-награды.
- `type` - тип предмета для матч-3. Предметы с одинаковым `type` можно собирать в одну линию, даже если `itemId` и `category` разные. Обязателен для `category: "match-3"` и `category: "rare_match-3"`.
- `damage` - сколько урона наносит матч-3 предмет. Обязателен для `category: "match-3"` и `category: "rare_match-3"`.
- `heal` - сколько прогресса добавляет в шкалу лечения игрока. Обязателен для `category: "match-3"` и `category: "rare_match-3"`.
- `aggression` - сколько агрессии добавляет врагу или бою. Обязателен для `category: "match-3"` и `category: "rare_match-3"`.
- `calm` - сколько успокоения добавляет матч-3 предмет. Обязателен для `category: "match-3"` и `category: "rare_match-3"`.
- `Leave_side` - угол вылета активированного матч-3 предмета при исчезновении, от `0` до `360`, где `0` означает вертикально вверх. Обязателен для `category: "match-3"` и `category: "rare_match-3"`.
- `death_time` - время исчезновения активированного матч-3 предмета в секундах. Во время исчезновения предмет растворяется и увеличивается до x2. Обязателен для `category: "match-3"` и `category: "rare_match-3"`.
- `createsOnFour` - необязательная явная ссылка на предмет, который создается при сборе 4 одинаковых `type`. Для обычных боевых предметов это усиленная версия.
- `createsOnFive` - необязательная явная ссылка на особый предмет, который создается при сборе 5+ одинаковых `type`. Сейчас этим предметом обычно является батарея; такие предметы не должны случайно падать на поле.
- `battleUse: "battery"` - специальное боевое поведение батареи. Такой предмет не собирается обычными match-3 комбинациями, но может быть объединен с соседней клеткой.
- `max_hp_modif` - необязательный пассивный боевой модификатор предмета в инвентаре: каждое количество предмета увеличивает максимальное здоровье игрока на указанное число.
- `heal_hp_modif` - необязательный пассивный боевой модификатор предмета в инвентаре: каждое количество предмета увеличивает размер здоровья, восстанавливаемого при срабатывании шкалы лечения.
- `modificate` - необязательный список боевых модификаторов от предмета в инвентаре. Каждый элемент указывает `itemId` матч-3 предмета и добавки к `damage`, `heal`, `aggression`, `calm`; добавки умножаются на количество этого предмета в инвентаре игрока.
- `transform_chance` - шанс в процентах за один предмет в инвентаре заменить случайно сгенерированный `transform_from_itemId` на `transform_to_itemId`.
- `transform_from_itemId` - какой матч-3 предмет должен был сгенерироваться.
- `transform_to_itemId` - какой предмет появится вместо него при успешном броске шанса.
- `nameTextKey` - ключ названия предмета в локалях.
- `descriptionTextKey` - ключ описания предмета в локалях. В текстах описаний можно вставлять иконки предметов через маркер `{item:itemId}`, например `{item:bullet}` или `{item:bullet_power}`. Размер отдельной иконки можно умножить относительно базового `inlineItemIconEm` маркером сразу после нее: `{item:bullet}\s{1.5}` сделает эту иконку в 1.5 раза больше базовой; в JSON-строке это пишется как `{item:bullet}\\s{1.5}`. Сейчас такой rich-text рендер поддерживается в tooltip предметов на карте и в бою, а также в описании ярости врага в боевом окне; если `itemId` не найден в каталоге, маркер остается обычным текстом.
- `icon` - обычная иконка из `data/Assets/item`.
- `bigIcon` - крупная иконка из `data/Assets/big_icon`.
- `showInHudAlways` - если `true`, строка всегда показывается в HUD.
- `hudOrder` - порядок отображения в HUD; меньшее число идет выше.
- `hudValue` - специальный источник значения HUD. Сейчас используются `health` и `experience`, потому что здоровье и опыт не лежат в обычном инвентаре.
- `battleTimeStopSeconds` - необязательное боевое значение для предмета часов: на сколько секунд останавливается таймер ярости при клике по предмету в бою. Пока таймер остановлен, слот часов затемняется, показывает обратный отсчет и не может быть активирован повторно.

Если новый `itemId` появляется в карте, награде, магазине или стартовом состоянии игрока, он должен быть добавлен в `data/settings/items.jsonc`. Валидатор перед START проверяет наличие предмета в каталоге и наличие всех `nameTextKey`/`descriptionTextKey` во всех активных локалях.

## Таблица опыта

Таблица опыта находится в `data/player/experience-table.jsonc`. Опыт игрока хранится в `playerState.experience.total` как общее количество опыта за забег, а текущий уровень хранится в `playerState.experience.level` и может пересчитываться по таблице. Повышение уровня не тратит и не обнуляет `total`: пороги уровня читаются как накопительный общий опыт.

Верхнее поле `limitedItemInventoryThreshold` задает общий порог для предметов из `data/settings/items.jsonc` с `limitByInventory: true`. Текущий порог - `50`: если у игрока уже есть 50 или больше такого предмета, он больше не появляется в выборе награды за уровень. В reward-наградах карты/врага и товарах магазина такой предмет остается видимым с красной плашкой MAX/МАКС и затемненной иконкой, но магазин не дает выбрать его для покупки, а reward-начисление пропускает его после кнопки `ui.claimReward`. Это не жесткий максимум инвентаря: уже выданные предметы не удаляются и прямое количество не обрезается.

Верхнее поле `rewardWeightReductionPercentPerInventoryItem` сглаживает выбор наград за уровень по текущему инвентарю. Для каждого предмета игра берет базовый `weight` из таблицы уровня и уменьшает его на этот процент за каждую штуку такого же `itemId` в инвентаре, затем округляет итог вверх. Текущее значение - `1`: если у предмета базовый вес `150`, а в инвентаре уже `31` такой предмет, итоговый вес будет `ceil(150 * (1 - 31 * 0.01)) = 104`. Если множитель ушел ниже нуля, итоговый вес считается `0`.

Поля уровня:

- `level` - номер уровня.
- `requiredExperience` - сколько общего опыта нужно накопить, чтобы достигнуть этого уровня.
- `eventImage` - картинка события повышения уровня из `data/Assets/events`.
- `textKey` - текст, который будет показан игроку при повышении.
- `rewardCount` - сколько уникальных вариантов показать игроку на выбор из массива `rewards`. Если вариантов меньше, чем `rewardCount`, показываются только доступные варианты.
- `rewards` - список возможных предметов за повышение уровня.

Поля награды уровня:

- `itemId` - любой предмет из `data/settings/items.jsonc`.
- `weight` - вес предмета при случайном выборе.
- `amount` - сколько предметов выдать.

Текущее количество предметов в наградах за уровень растет блоками по 10 уровней: 1-10 дают по 1 предмету, 11-20 - по 2, 21-30 - по 3, 31-40 - по 4, 41-50 - по 5.

Веса выпадения предметов за уровень идут циклом из 5 уровней. Уровни с фазой 1-2 используют первый набор весов: обычные нож/пуля/граната по `2125`, их усиленные версии по `1025`, обычные бинт/щит по `1600`, усиленные бинт/щит по `700`, `red` и `green` по `1180`. Уровни с фазой 3-4 используют второй набор: обычные нож/пуля/граната по `1600`, их усиленные версии по `700`, обычные бинт/щит по `2095`, усиленные бинт/щит по `1005`, `red` и `green` по `1180`. Уровень с фазой 5 использует третий набор: обычные нож/пуля/граната/бинт/щит по `400`, все усиленные версии по `1000`, `red` и `green` по `2400`. Затем цикл повторяется: 6-7 снова первый набор, 8-9 второй, 10 третий.

Когда игрок получает опыт и пересекает один или несколько порогов `requiredExperience`, игра после текущего окна награды открывает окно повышения уровня. Для каждого достигнутого уровня варианты сначала фильтруются по `limitByInventory`, затем выбираются случайно по `weight` без повторов. Игрок выбирает один вариант, и только выбранный предмет начисляется в инвентарь. Если за один раз достигнуто несколько уровней, окна выбора показываются по очереди.

Battle-вариант может передать учебному повышению уровня объект `levelUpTutorial` с `enabled: true`, общим `introTextKey` и массивом `rewards[]`, где каждая запись содержит `itemId` и поясняющий `textKey`. Такой объект действует только на порог, пересечённый опытом из этой battle-награды: варианты берутся в указанном порядке, а выбор блокируется до знакомства с каждой карточкой. Это позволяет обучать стандартному level-up UI, не меняя глобальные `rewardCount`, веса и пулы `experience-table.jsonc`.

HUD опыта использует предмет `exp` из `data/settings/items.jsonc` и всегда показывает строку сразу после здоровья. Значение отображается как `общий опыт/опыт до следующего уровня`, например `35/50`. После перехода порога общий опыт не сбрасывается: `55` опыта после порога `50` будет показано как `55/105`, а не как остаток текущего уровня. Если следующий уровень отсутствует, вместо чисел показывается локализованный ключ `ui.maxLevel`.

## Генерация карты

Для каждого элемента `levels[]` создается отдельный уровень карты. Количество точек берется из `nodes.count`, если нужно фиксированное значение, или случайно из диапазона `nodes.min`-`nodes.max`.

Генерация карты детерминирована, если передан seed. `src/map/map-run-controller.js` создает `run` seed для забега и производный `map:<mapId>:<campaignIndex>` seed для конкретной карты, затем вызывает `generateMap(config, { random })`. В лог событий пишутся оба значения, поэтому баг карты можно воспроизвести через URL вида `index.html?seed=A7K9M2QX4T8ZB3NC`.

Игровые выборы карты, которые происходят уже после генерации графа, тоже используют производные seed от текущей карты: reward-пулы получают домен `reward:<source>:<campaignIndex>:<nodeId>:<eventName>`, а варианты level-up получают домен `level-up:<campaignIndex>:<level>:<requiredExperience>`. Это значит, что декоративная случайность карты, например птицы и интервалы ambient-анимации, не должна менять выпавшие награды или варианты повышения уровня при том же `run` seed.

Боевые события получают производный seed с именем `battle:<mapId>:<nodeId>:data/enemy/<enemyId>.jsonc:<attempt>`. Этот seed записывается в лог карты перед открытием боя и в лог самого боя, а в `BattleRequest` дополнительно передается `enemyConfigUrl`. Боевой модуль создает из seed RNG для стартового поля, резервного поля, препятствий, refill, shuffle, gold-замены и random-target эффектов ярости. Декоративные визуальные эффекты боя остаются на отдельном несидированном `Math.random()`, чтобы частицы и снаряды не меняли игровой результат.

Боевой экран параллельно с человекочитаемым логом ведет `battleTrace`: JSON-снимок попытки боя. В нем есть версия trace-формата, версия приложения, `seed`, `seedName`, `enemyConfigUrl`, технический `enemyId`, необязательный `baseEnemyId`, стартовое состояние доски/резерва/препятствий, список игровых действий игрока и финальный outcome. В окне боевого лога кнопка `ui.downloadBattleTrace` скачивает этот JSON для разбора и будущего replay/debug-инструмента. Это экспорт сырья для воспроизведения, а не готовый проигрыватель боя.

Для каждой точки:

1. Сначала уровень создает обязательные события из `levels[].events`, где `guaranteed: true`. Одна запись создает одну обязательную точку; если нужно несколько одинаковых обязательных точек, запись можно повторить.
2. Оставшиеся точки уровня добираются случайно из событий этого же `levels[].events`, у которых `guaranteed` не включен и `weight` больше `0`.
3. `name` ищется в общем каталоге `events`, где уже лежат `type` и `icon`.
4. `type` определяет игровую логику, `icon` сохраняется как иконка точки.
5. Из соответствующего раздела карты по `type` выбирается payload, у которого `eventName` совпадает с именем события точки и подходит по `minLevel`/`maxLevel`.
6. Если payload-вариантов несколько, используется их внутренний `weight`.

Валидатор перед стартом требует, чтобы каждый случайный или гарантированный тип события имел подходящий payload-вариант для своего уровня. Если событие включено на уровне, но для этого уровня нет payload по `minLevel`/`maxLevel`, забег не начнется и игра покажет ошибку данных.

Событие `dialog` показывает отдельное диалоговое окно поверх карты: фон карты затемняется и размывается по настройкам `data/settings/map-ui.jsonc`, в центре показывается персонаж, ниже текст и кнопки ответов. Payload диалога хранит конкретную постановку сцены: `characterImage`, `characterWidthPct`, `characterCenterXPct`, `characterCenterYPct`, `initialStepId` и массив `steps`. `characterWidthPct` задает ширину персонажа в процентах ширины экрана: `100` значит во всю ширину. `characterCenterXPct` и `characterCenterYPct` задают центр персонажа в процентах ширины/высоты экрана. Для старых диалогов еще поддерживаются `characterWidthPx` и `characterBottomPx`, но новый понятный формат - проценты ширины и координаты центра по X/Y. У шага есть `stepId`, `textKey` и `answers`. Ответ должен делать ровно одно действие: `nextStepId` продолжает диалог, `eventName` запускает любое событие из общего каталога карты, `end: true` завершает диалог. Точка считается пройденной после конца диалога или после завершения события, на которое сослался ответ.

На первом уровне `WildWest` создаются две стартовые точки: прямой бой `battle_easy` с `easy_Foxy` и диалог `opening_dialog` со Свити. Игрок выбирает один из этих стартов, как обычную точку первого уровня. Диалог использует существующий `dialog`-контракт и предлагает четыре перехода в события первого уровня: `battle_test` с врагом `test`, `shop`, `heal` и `reward`. Для таких переходов у целевого события должен быть payload-вариант на том же уровне, иначе валидатор данных остановит запуск вместо того, чтобы диалог вел игрока в пустоту.

Магазин первого уровня, в который ведет ответ Свити `shop`, продает пул предметов из наград за повышение уровня: обычные боевые предметы (`item_Knife`, `item_bullet`, `item_granate`, `item_Bandage`, `item_Shield`) стоят 50 золота, усиленные версии (`item_Knife_power`, `item_bullet_power`, `item_granate_power`, `item_Bandage_power`, `item_Shield_power`) стоят 100 золота, `red` стоит 200 золота, а `green` стоит 100 золота. Это отдельный стартовый ассортимент; поздние магазины карты настраиваются своими payload-вариантами.

Текст диалога печатается по буквам. Скорость задается в `data/settings/map-ui.jsonc` через `dialog.textLetterMs` (по умолчанию 100 мс на букву). Клик по сцене во время печати сразу раскрывает текущую фразу, но после этого сцена больше не выбирает ответ и не закрывает окно: для перехода к следующей реплике, запуска связанного события или завершения любого диалога игрок обязан нажать одну из кнопок собственного ответа. Клик по самой кнопке ответа не всплывает к сцене и не раскрывает следующую реплику. Варианты ответов появляются только после полной фразы; длительность их появления задается через `dialog.answersFadeMs` (по умолчанию 1000 мс). При наведении мышью или клавиаточном фокусе выбранная кнопка ответа загорается жёлтым и получает светящийся контур.

После генерации точек строятся дороги:

- Каждая точка текущего уровня получает хотя бы одну связь вперед.
- Каждая точка следующего уровня получает хотя бы одну связь назад.
- Если в карте есть `pathRules.connectionCountWeights`, каждая точка выбирает желаемое количество исходящих дорог по весам. Например `1/20`, `2/65`, `3/15` означает: иногда один путь, чаще два пути, иногда три пути. Это общий стиль карты, чтобы не получать длинные коридоры без развилок.
- `levels[].paths.connectionCountWeights` может переопределить это правило для конкретного уровня.
- `pathRules.maxSinglePathChain` ограничивает длинные коридоры: если маршрут уже несколько раз подряд шел через точки с единственным выходом, генератор начинает пытаться дать следующей точке хотя бы две дороги. Это считается в одиночных переходах, а не в визуальных точках: значение `2` обычно значит, что один путь может тянуться примерно через 2-3 точки, но не через половину карты. Уровень может переопределить это через `levels[].paths.maxSinglePathChain`.
- Без `connectionCountWeights` генератор использует старое правило: `levels[].paths.minConnectionsFromNode` как минимальное желаемое количество исходящих дорог.
- `levels[].paths.maxConnectionsFromNode` ограничивает желаемое количество дорог из `connectionCountWeights` и дополнительные связи. Если нужно иногда получать 3 пути, у уровня должен быть `maxConnectionsFromNode: 3`.
- `levels[].paths.extraConnectionChance` добавляет шанс на дополнительные связи после обязательных дорог. Если шанс `0`, карта старается оставить только минимальный читаемый маршрутный скелет: выходы текущего уровня и входы следующего.
- Если финальный уровень содержит boss-событие, предыдущий уровень ведет к нему обычными дорогами, как к любой другой точке.

При создании дорог генератор старается не добавлять связь, если она пересекает уже выбранные дороги между теми же соседними уровнями. Если без пересечения нельзя сохранить обязательное правило "каждая точка имеет вход/выход", связь все равно добавляется: проходимость карты важнее идеальной геометрии.

После базовой раскладки каждая обычная точка получает стабильный визуальный разброс по X/Y, причем горизонтальный разброс заметнее вертикального. Разброс вычисляется от `mapId` и `node.id`, поэтому точки не прыгают при перерисовке, смене языка или обновлении HUD. Диапазон разброса теперь задается в `data/settings/map-ui.jsonc` через `nodes.positionJitterPct.x/y`. Горизонтальный разброс автоматически ограничивается внутри своей позиции, чтобы точка не перескочила через соседнюю и не создала новые пересечения дорог.

## Правила движения

На первом уровне игрок может выбрать любую точку.

После выбора точки:

- текущая точка становится пройденной;
- доступными становятся только точки, указанные в `connectedTo`;
- путь, по которому игрок пришел, навсегда подсвечивается зеленым;
- пути от текущей точки к доступным следующим точкам подсвечиваются желтым;
- остальные пути остаются бледными.

После завершения события viewport плавно прокручивается к новым доступным точкам примерно за 1 секунду. Для `skip` это происходит после закрытия диалога, для `shop` и `heal` - после ухода, покупки или лечения. Цель прокрутки - разместить доступные точки примерно в нижней трети экрана, если это возможно по границам карты. Это сделано, чтобы игрок мог выбирать следующий шаг без ручного скролла вверх-вниз после каждого события.

Точки отображаются крупными иконками из общего каталога `events[].icon`, поэтому визуал события на карте настраивается прямо в JSONC карты. Boss-точка теперь тоже берёт иконку из `events[].icon`, обычно `data/Assets/icons/boss.png`. Подписей у точек нет, чтобы карта не шумела.

Минимальная кликабельная область обычной точки составляет `124x124` пикселя, её иконка — `80x80`. Для boss-точки используются `152x152` и `104x104` пикселя соответственно. Вертикальный шаг между центрами соседних уровней рассчитывается из прежней раскладки и умножается на `3`; верхний и нижний безопасные отступы при этом остаются прежними, чтобы вступительная прокрутка по-прежнему завершалась на видимых точках первого уровня.

Если у карты включен `firstNodeArrow`, стрелка показывается только у первой доступной точки до начала её события. Она расположена справа под углом 45 градусов и указывает на край иконки, не пересекаясь с её максимальным hover-размером. После клика стрелка исчезает и не переносится на следующие уровни.

## Событие Skip

`skip` - тестовое событие.

Поля:

- `minLevel`, `maxLevel` - диапазон уровней.
- `textKey` - ключ текста в локали.
- `weight` - вес конкретного варианта.

При активации показывает текст из локали. Сейчас это используется для быстрых проверок прохождения карты.

## Событие Shop

`shop` открывает затемненный экран торговца.

Поля события:

- `minLevel`, `maxLevel` - диапазон уровней.
- `weight` - вес варианта торговца.
- `dialogTextKey` - ключ реплики торговца.
- `eventImage` - изображение события из `data/Assets/events`, сейчас используется `data/Assets/events/trader.png`.
- `items` - список торговых позиций.

Поля торговой позиции:

- `itemId` - предмет, который покупает игрок.
- `amount` - сколько предметов продается одной позицией.
- `goldPrice` - цена всей позиции в золоте.

Игрок может выбрать одну или несколько позиций, но каждую позицию только один раз. Если товар ограничен через `limitByInventory` и текущее количество уже достигло `limitedItemInventoryThreshold`, карточка остается в магазине с красной плашкой MAX/МАКС и затемненной иконкой, но выбрать и купить ее нельзя. Перед покупкой появляется внутреннее подтверждение в интерфейсе игры. Если золота не хватает, показывается `ui.notEnoughGold`, окно магазина не закрывается.

## Событие Heal

`heal` открывает затемненный экран лекаря.

Поля события:

- `minLevel`, `maxLevel` - диапазон уровней.
- `healPercent` - сколько процентов от максимального HP восстановить.
- `dialogTextKey` - ключ реплики лекаря.
- `eventImage` - изображение события из `data/Assets/events`, сейчас используется `data/Assets/events/doctor.png`.
- `goldPrice` - цена лечения в золоте. Если `0`, лечение бесплатное.
- `weight` - вес варианта лечения.

В окне лечения показывается:

- персонаж справа;
- реплика персонажа;
- сколько HP будет восстановлено;
- текущее и максимальное HP;
- кнопка лечения;
- кнопка ухода.

Если золота не хватает, показывается `ui.notEnoughGold`, окно лечения не закрывается. После успешного лечения или ухода игрок возвращается на карту.

## Заготовка Battle

`battle` пока не запускает полноценный бой, но данные уже описываются в карте.

Поля:

- `enemyId` - идентификатор врага.
- `minLevel`, `maxLevel` - диапазон уровней.
- `weight` - вес врага.

Задник любого боя, включая boss-событие, задается в конкретном battle-варианте через `background`. Враги и боссы используют изображения из `data/Assets/enemy`.

## Данные врагов

Джейсонки врагов хранятся в `data/enemy`. Канонический путь для врага: `data/enemy/<enemyId>.jsonc`. Карта ссылается на врага через `enemyId`, и этот же технический id должен лежать внутри файла в поле `enemyId`. Проверка данных перед START заранее загружает активные файлы врагов из кампании, проверяет совпадение `enemyId` с именем файла без `.jsonc` и проверяет минимальный боевой контракт.

Главные поля врага:

- `enemyId` - технический идентификатор конфига врага. Должен совпадать с именем файла без `.jsonc` и с `enemyId` в карте.
- `baseEnemyId` - необязательная связь с базовым персонажем для вариантов сложности или boss-вариантов, например `easy_Foxy` может иметь `baseEnemyId: "Foxy"`. Это поле нужно для логов, редактора, статистики и группировки; видимое имя по-прежнему задается через `nameTextKey`.
- `nameTextKey` - ключ имени врага в локали.
- `descriptionTextKey` - ключ описания врага в локали.
- `stageCount` - ожидаемое количество стадий.
- `wall` - необязательное общее количество стен между клетками боя. `0` или отсутствие поля означает бой без стен.
- `box` - необязательное общее количество коробок, закрывающих клетки боя. `0` или отсутствие поля означает бой без коробок.
- `vines` - необязательное общее количество лиан, накрывающих клетки боя. `0` или отсутствие поля означает бой без лиан.
- `stages` - стадии врага по порядку.
- `reward` - награда после победы.

Поля стадии:

- `stageId` - уникальный id стадии внутри врага.
- `nameTextKey` - ключ имени стадии.
- `appearance` - внешний вид стадии, путь в `data/Assets/enemy`.
- `health` - сколько урона нужно нанести, чтобы перейти к следующей стадии.
- `shield` - необязательный щит стадии. Каждый активированный предмет с уроном снимает `1` щит вместо урона по `health`; когда щит кончается, оставшиеся предметы той же активации наносят обычный урон.
- `wall` - необязательное количество стен только для этой стадии. Если поле указано, оно переопределяет общий `wall` врага.
- `box` - необязательное количество коробок только для этой стадии. Если поле указано, оно переопределяет общий `box` врага.
- `vines` - необязательное количество лиан только для этой стадии. Если поле указано, оно переопределяет общий `vines` врага.
- `convert` - необязательные пассивные превращения новых предметов только для этой стадии. Корневой `convert` у врага больше не используется, чтобы стадии могли иметь разную логику генерации.
- `itemStatModifiers` - необязательные множители боевых параметров активированных предметов только для этой стадии. Селектор `itemTypes` выбирает все предметы с указанным `type`, включая усиленные варианты; `itemId`/`itemIds` выбирает точные `itemId`. Поддержанные множители: `damage`, `heal`, `aggression`, `calm`.
- `aggression.threshold` - сколько агрессии нужно накопить до атаки врага.
- `aggression.damage` - сколько урона враг наносит при достижении порога агрессии.
- `aggression.resetOnTrigger` - старое поле для будущей настройки; в текущем MVP агрессия после атаки всегда сбрасывается в `0`.
- `rage.secondsToUltimate` - время в секундах до ульты.
- `rage.resetAfterUltimate` - сбрасывать ли таймер ярости после ульты.
- `rage.targetTypes` - запасные типы предметов для визуальной ярости, если у эффекта нет собственных целей или у ульты пока нет `effects`. Тип берется из `data/settings/items.jsonc`.
- `rage.targetItemIds` - запасные конкретные `itemId` предметов для визуальной цели ярости, если нужна привязка не к типу, а к отдельному предмету.
- `ultimate` - специальная способность стадии.

Локализованное описание `ultimate.descriptionTextKey` выводится в боевой панели как rich-text: поддерживает переносы строк и иконки предметов через `{item:itemId}`. Текущая договоренность для врагов - писать его в две смысловые строки: `Ярость: ...` описывает эффект ульты, `Особенное: ...` описывает только важные открытые правила боя: появление нестандартного объекта через "в бою может появиться {item:itemId}", стартовые препятствия/щит и изменения характеристик предметов. Нестандартным считается то, что не входит в обычный случайный добор `category: "match-3"`: например `rare_match-3` усиленные предметы, бочки и особый мусор. Обычные предметы вроде `Bandage`, `Shield`, `trash_1` и `trash_2` отдельно не объявляются только потому, что враг может чаще их подмешивать. Не нужно раскрывать точные источники `convert`, шансы и цепочки усиления вроде "оружие становится бинтом" или "граната усиливается"; это игрок изучает сам, как и положено в игре, а не в налоговой форме. Размер иконок в этой боевой панели фиксируется CSS-контекстом `.battle-scaffold-ultimate`, а не локальными `\s{...}` множителями в тексте: так публикация на GitHub Pages и локальный `localStorage` не дают разный масштаб. Это не отдельные поля JSON, а читаемый UI-текст; механику по-прежнему задают `rage`, `ultimate.effects`, `convert`, `itemStatModifiers`, `shield`, `wall`, `box` и `vines`.

В текущем MVP боя щит врага стартует из `stage.shield`, а при переходе на следующую стадию обновляется из новой стадии. Щит считается по количеству активированных повреждающих предметов, а не по сумме урона: если у врага `3` щита и активировано `5` предметов с уроном, первые `3` снимут щит, а оставшиеся `2` нанесут свой обычный урон по `health`. Эффекты ярости могут восстановить щит выше стартового значения стадии, но не выше `data/settings/battle-ui.jsonc` -> `limits.enemyShieldMax` (по умолчанию `99`). Визуальная иконка щита берется из `data/settings/battle-ui.jsonc` -> `icons.enemyShield`.

В текущем MVP боя агрессия при достижении порога наносит игроку `aggression.damage` и сбрасывается в `0`, даже если пришло переполнение. Ярость каждую секунду уменьшает таймер, даже если на поле сейчас идет swap, падение, добор, каскад или ручное перемешивание. Если таймер дошел до `0`, пока поле занято, ярость ставится в ожидание, таймер остается на `0`, а сама ульта запускается только после завершения текущих действий и анимаций поля. При запуске ярость блокирует поле, запускает белую волну по иконкам предметов и выполняет `ultimate.effects` строго сверху вниз: анимация и применение одного эффекта завершаются до начала следующего. Каскады не проверяются между эффектами ульты; поле проверяется на комбинации только после завершения всей цепочки эффектов. После этого поле снова активно, таймер перезапускается от `rage.secondsToUltimate`.

Пассивное поле стадии `convert` работает только для новых предметов: при стартовой генерации, доборе и пополнении резервного поля во время этой стадии. Оно не превращает уже лежащие на поле предметы задним числом. `chance` задается долей от `0` до `1`, где `0.2` значит 20%, а `1` значит 100%. `from.itemId`/`from.itemIds` выбирает точные `itemId`, `from.itemTypes` выбирает все предметы с указанным `type`. `to.itemId` или `to.itemIds` задает возможные результаты; если результатов несколько, для каждого нового предмета выбирается случайный результат. Вражеский `stage.convert` проверяется раньше трансформаций инвентаря игрока: если стадия врага превратила новый предмет, игрок уже не улучшает именно этот предмет; если шанс стадии не сработал или правило не подходит, тогда проверяются `transform_chance` предметов игрока. Например, такой блок внутри конкретной стадии дает новым `trash_1` и `trash_2` шанс 20% стать `radioactive`:

```jsonc
"convert": [
  {
    "effectId": "convertItems",
    "type": "convertItems",
    "chance": 0.2,
    "from": {
      "itemId": ["trash_1", "trash_2"]
    },
    "to": {
      "itemId": ["radioactive"]
    },
    "scope": "all"
  }
]
```

`itemStatModifiers` текущей стадии применяется в момент активации предмета после базовых значений из `items.jsonc` и после инвентарных `modificate`. Множитель `0.8` означает -20%, `1.2` означает +20%, `0` полностью гасит выбранный параметр; итоговое значение не уходит ниже `0`. После всех добавок и множителей итоговые `damage`, `heal`, `aggression`, `calm` округляются вниз до десятых: например, `4.15` станет `4.1`. Это влияет только на активированные параметры match-3 предметов: обычные совпадения, каскады, череп, батарея и другие действия используют один расчет. Пассивный `dmgperturn`, счетчики предметов в `ultimate.effects` и генерация поля этим блоком не меняются. Если для батареи или черепа действие подавляет агрессию, множитель `aggression` все равно посчитается, но итоговая агрессия не начислится из-за самого действия.

```jsonc
"itemStatModifiers": [
  {
    "itemTypes": ["granate"],
    "multipliers": {
      "damage": 0.8,
      "aggression": 1.2
    }
  },
  {
    "itemTypes": ["Bandage"],
    "multipliers": {
      "heal": 0.8
    }
  }
]
```

Поддержанный эффект ульты `convertItems` преобразует предметы на поле:

```jsonc
{
  "effectId": "convertItems",
  "type": "convertItems",
  "from": {
    "itemTypes": ["junk_1", "junk_2", "junk_3"]
  },
  "to": {
    "itemIds": ["Bandage", "granate"]
  },
  "scope": "all"
}
```

`from.itemTypes` ищет все предметы указанного типа, поэтому в выборку попадут обычные и усиленные предметы с одинаковым `type`. `from.itemIds` или `from.itemId` выбирает конкретные `itemId` и не трогает предметы того же типа с другим id. `to.itemId` задает один предмет-замену, а `to.itemIds` задает список возможных замен; для каждой найденной клетки случайно выбирается один предмет из списка. Сейчас `scope` документирует намерение и используется как `all`: все найденные предметы заменяются после визуальной сцены ярости. Во время срабатывания ярости вокруг предметов, которые будут преобразованы этим эффектом, крутятся золотые светлячки. Сразу после преобразования поле проверяется на обычные match-3 комбинации; если они появились, они отрабатывают через общий каскадный пайплайн с эффектами, удалением, падением и добором.

Поддержанный эффект ульты `damagePlayerByBoardItems` наносит игроку урон по количеству предметов на поле:

```jsonc
{
  "effectId": "damagePlayerByBoardItems",
  "type": "damagePlayerByBoardItems",
  "count": {
    "itemTypes": ["granate"]
  },
  "modifier": 1
}
```

`count.itemTypes` считает все предметы указанных типов, включая усиленные предметы с тем же `type`. `count.itemIds` или `count.itemId` считает только конкретные `itemId`. Урон равен `количество найденных предметов * modifier`. Красные светлячки летят от найденных предметов к иконке здоровья игрока, после чего отрабатывает стандартная анимация изменения здоровья.

Поддержанный эффект `damagePlayerFixed` наносит заданный фиксированный урон независимо от состава поля:

```jsonc
{
  "effectId": "fixedStrike",
  "type": "damagePlayerFixed",
  "amount": 1000
}
```

Красные светлячки летят от врага к здоровью игрока. Поле `amount` должно быть неотрицательным числом; фактическое здоровье не уходит ниже нуля.

Поддержанный эффект ульты `HealingEnemyByBoardItems` лечит врага по количеству предметов на поле:

```jsonc
{
  "effectId": "HealingEnemyByBoardItems",
  "type": "HealingEnemyByBoardItems",
  "count": {
    "itemTypes": ["Bandage"]
  },
  "modifier": 1
}
```

`count.itemTypes` считает все предметы указанных типов, включая усиленные предметы с тем же `type`. Для текущих бинтов тип называется `Bandage`, с большой буквы, потому что именно так он записан в `data/settings/items.jsonc`. `count.itemIds` или `count.itemId` считает только конкретные `itemId`. Лечение равно `количество найденных предметов * modifier`, здоровье врага не поднимается выше максимума стадии, а зеленые светлячки летят от найденных предметов к иконке здоровья врага перед анимацией `+N`.

Поддержанный эффект ульты `RestoreEnemyShieldByBoardItems` восстанавливает щит врага по количеству предметов на поле:

```jsonc
{
  "effectId": "RestoreEnemyShieldByBoardItems",
  "type": "RestoreEnemyShieldByBoardItems",
  "count": {
    "itemTypes": ["Shield"]
  },
  "modifier": 1
}
```

`count.itemTypes` считает все предметы указанных типов, включая усиленные предметы с тем же `type`. Для текущих щитов тип называется `Shield`, с большой буквы. `count.itemIds` или `count.itemId` считает только конкретные `itemId`. Восстановление равно `количество найденных предметов * modifier`, текущий щит врага не поднимается выше `limits.enemyShieldMax` из `data/settings/battle-ui.jsonc`. Синие светлячки летят от найденных предметов к самой иконке щита врага перед анимацией `+N`; если щита перед этим не было, иконка появляется сразу перед полетом светлячков.

Ульта хранит `nameTextKey`, `descriptionTextKey` и массив `effects`. Порядок записей в массиве важен: если сначала стоит `convertItems`, потом `damagePlayerByBoardItems`, потом еще один `convertItems`, то урон считается уже по результату первого преобразования, а последнее преобразование происходит только после анимации урона. Массив нужен, чтобы одна ульта могла делать несколько вещей и чтобы список типов эффектов можно было расширять без переделки формата. `descriptionTextKey` должен объяснять игроку не только саму ярость, но и важные пассивные особенности врага. При этом он не должен превращаться в распечатку JSON: показываем нестандартные объекты и изменения характеристик, но не раскрываем точные правила конверта и усиления.

Текущие типы эффектов в формате:

- `convertItems` - поддержано в коде; превращает элементы поля в другие элементы.
- `damagePlayerByBoardItems` - поддержано в коде; наносит игроку урон: количество найденных элементов на поле умножается на `modifier`.
- `damagePlayerFixed` - поддержано в коде; наносит игроку фиксированный урон из `amount`, не зависящий от поля.
- `HealingEnemyByBoardItems` - поддержано в коде; лечит врага: количество найденных элементов на поле умножается на `modifier`.
- `RestoreEnemyShieldByBoardItems` - поддержано в коде; восстанавливает щит врага: количество найденных элементов на поле умножается на `modifier`.
- `kamikaze` - поддержано в коде; игрок получает урон, равный текущему здоровью врага, затем враг получает самоурон на то же значение. Щит врага самоурон не поглощает.
- `healSelfByBoardItems` - зарезервировано на будущее; старое имя, сейчас не запускает лечение.
- `freezeRandomItems` - зарезервировано на будущее; замораживает случайные элементы.

Копируемые примеры блоков `rage`, `ultimate.effects`, цепочек эффектов и пассивного `convert` лежат в `data/enemy/rage.example.jsonc`. Это справочный файл, игра его напрямую не загружает.

Для выбора элементов используются селекторы:

- `itemIds` / `itemId` - конкретные `itemId` из `data/settings/items.jsonc`; это точный выбор предметов по имени id.
- `itemTypes` - типы матч-3 предметов из `data/settings/items.jsonc`, например `granate`, `Bandage`, `junk_1`; это выбор всех предметов с таким типом, включая усиленные варианты.

Награда врага повторяет общий формат наград карты:

- `reward.itemCount` - сколько записей взять из массива `reward.rewards`.
- `reward.eventImage` - изображение для общего reward-окна после победы.
- `reward.dialogTextKey` - ключ текста, который показывается в reward-окне после победы.
- `reward.rewards` - пул наград за победу.
- `type: "health"` с `amount` - восстановление здоровья игрока после победы.
- `type: "experience"` с `amount` - опыт игрока.

Награда врага показывается тем же окном, что и событие `reward` на карте. Важно: победа в бою сама по себе еще не завершает точку карты. Сначала карта показывает награду, затем после кнопки `ui.claimReward` начисляет ее, помечает battle-точку пройденной и открывает следующие точки.

## Событие Reward

`reward` показывает золото или предметы из списка наград в отдельном окне награды, но начисляет их только после нажатия кнопки `ui.claimReward`. Если предмет ограничен через `limitByInventory` и у игрока уже есть не меньше `limitedItemInventoryThreshold`, он все равно показывается в reward-окне с красной плашкой MAX/МАКС и затемненной иконкой, но при нажатии `ui.claimReward` не начисляется. `eventImage` открывается четким изображением внутри окна, примерно 1 секунду остается резким, затем за 2 секунды становится размытым фоном. В центре появляются крупные иконки полученных предметов: сами иконки стартуют маленькими и за 4 секунды увеличиваются до нормального размера. Подложка с названием и количеством появляется только после завершения зума иконки. Снизу выводится текст события и кнопка `ui.claimReward`.

Поля:

- `minLevel`, `maxLevel` - диапазон уровней.
- `weight` - вес варианта награды.
- `dialogTextKey` - ключ текста события награды.
- `eventImage` - изображение события из `data/Assets/events`, например `reward_gold.png`, `reward_skull.png`, `reward_swap.png`, `reward_clock.png`.
- `itemCount` - сколько записей случайно выбрать из массива `rewards`.
- `rewards` - пул возможных наград.

Типы наград:

- `gold` с `amount`.
- `experience` с `amount`.
- `health` с `amount`.
- `item` с `itemId` и необязательным `amount`; если `amount` не указан, выдается `1`.

Шансов внутри `rewards` сейчас нет: после выбора варианта награды игра случайно берет `itemCount` записей из массива `rewards` без повторов. Поэтому `itemCount` нужен, если один reward-вариант должен быть пулом вроде "выдать 1 предмет из 3". Если вариант всегда должен выдавать все перечисленные награды, `itemCount` можно считать лишним и позже заменить правилом "выдать весь список".

## Boss

Босс больше не создается кодом как отдельная системная точка с `eventType: "boss"`.
Актуальный формат: босс описывается как обычное событие из `events[]` с `type: "battle"`,
собственным `name`, например `boss`, и иконкой `data/Assets/icons/boss.png`.

Поля:

- `events[]` - добавить событие с `name: "boss"`, `type: "battle"` и boss-иконкой.
- `levels[]` - на финальном уровне добавить ссылку на `name: "boss"`.
- `battle[]` - добавить payload для `eventName: "boss"` с `enemyId`, `background`, `minLevel`, `maxLevel` и `weight`.
- `nodeTitleTextKey` - необязательный ключ названия финальной точки, если нужно показать имя босса.

Карта завершается по общему правилу: если пройденная точка не имеет `connectedTo`,
игра выполняет `onComplete` текущей карты из `data/settings/campaign.jsonc`.

## Состояние игрока

Стартовое состояние находится в `data/player/default-player-state.json`.

Основные разделы:

- `health.current` - текущее здоровье.
- `health.max` - максимальное здоровье.
- `experience.level` - текущий достигнутый уровень игрока.
- `experience.total` - общее количество опыта игрока за забег.
- `heal.health` - сколько здоровья восстановить, когда шкала лечения достигает максимума.
- `heal.current` - текущий прогресс шкалы лечения.
- `heal.max` - максимум шкалы лечения. После срабатывания шкала уменьшается на это значение.
- `inventory` - инвентарь игрока.

## Чит-команды игрока

Конфиг чит-команд находится в `data/player/cheats.json`. Команды являются сырыми последовательностями клавиш, поэтому не проходят через локали: если в JSON написано `iddqd`, игрок вводит именно `iddqd`, а не перевод, синоним или надежду на автокоррекцию.

Поля конфига:

- `enabled` - общий выключатель читов. Если `false`, ввод команд игнорируется.
- `inputMode` - сейчас поддерживается только `"typedSequence"`: игра собирает короткий буфер набранных символов и проверяет, чем он заканчивается.
- `bufferMaxLength` - максимальная длина буфера ввода. Значение должно покрывать самую длинную команду.
- `activation.scope` - где можно активировать читы. Сейчас поддерживается `"mainMenu"`.
- `activation.command` - команда активации. В текущем конфиге это `iddqd`.
- `activation.sound` - звук подтверждения активации. Он предзагружается вместе с остальными ассетами и проигрывается через общую громкость эффектов `soundVolume`.
- `testRun.enabled` - включает скрытый запуск QA-карты после активации читов.
- `testRun.buttonTextKey` - ключ локали для скрытой кнопки в главном меню.
- `testRun.mapId` и `testRun.config` - id и путь к JSONC test-run карты. Сейчас используется `SmokeTest` и `data/maps/SmokeTest.jsonc`.
- `testRun.playerState` - отдельный стартовый player-state для test-run. Он валидируется тем же контрактом, что и обычный `default-player-state.json`.
- `testRun.mapNameTextKey`, `testRun.victoryTitleTextKey`, `testRun.victoryMessageTextKey` - необязательные ключи локали для логов и завершения test-run карты.
- `commands[].id` - внутриигровой идентификатор действия. Сейчас используются `autoWin` и `levelUp`.
- `commands[].scope` - экран, на котором команда работает: `"battle"` или `"map"`.
- `commands[].command` - набираемая последовательность для этого действия.

Текущее поведение:

- В главном меню ввод `iddqd` активирует читы на текущую browser-сессию и проигрывает звук выстрела из `activation.sound`.
- После активации в главном меню появляется скрытая кнопка `testRun`, если этот блок включен и валиден. Она запускает SmokeTest-карту отдельным run и не добавляет ее в обычную кампанию.
- На карте после активации ввод `lvl` добавляет игроку ровно столько опыта, сколько нужно до следующего порога уровня, и открывает обычное окно выбора награды за уровень.
- В бою после активации ввод `win` завершает бой обычной победой через стандартный flow `BattleResult`, награды врага и прогресс карты.

Ввод читов игнорируется, если фокус находится в `input`, `textarea`, `select` или `contenteditable`, чтобы служебная команда не срабатывала во время обычного редактирования текста. Читы не используют `window.__wildwestDebug`: debug-хук остается инструментом проверки, а не игровым API.

Свойства матч-3 предметов больше не лежат в состоянии игрока. Они задаются в `data/settings/items.jsonc` у предметов с `category: "match-3"` и `category: "rare_match-3"` через `type`, `damage`, `heal`, `aggression` и `calm`. Обычные `match-3` предметы могут случайно падать сверху, а `rare_match-3` предметы не входят в случайный добор и должны появляться через бонусы или другие эффекты. При этом оба типа категорий участвуют в совпадениях по общему `type`, кроме предметов со специальным `battleUse: "battery"`. Связи бонусов задаются там же: `createsOnFour` для усиленного предмета и `createsOnFive` для генератора. Необязательное поле `sound_effect` задает звук активации предмета, например `data/Assets/sound/granate.mp3`; если поля нет, предмет активируется без звука. Звук проигрывается только при фактической активации предмета в бою и использует общую громкость `soundVolume`.

Поле `dmgperturn` у match-3 предмета задает пассивный урон игроку за обычный swap. После каждого принятого обычного обмена соседних клеток, который не использует череп, перчатку, батарею, ручное перемешивание или каскад, игра считает все такие предметы на видимом поле и наносит `количество * dmgperturn` урона. Например, если на поле лежат 3 `radioactive` с `dmgperturn: 1`, обычный swap нанесет игроку 3 урона. Клетки под коробками не считаются: предмет под коробкой считается неактивным и не наносит пассивный урон. Предметы под лианами считаются активными. Каскады и специальные предметы этот урон не запускают. Красные светлячки этого урона летят именно от предметов с `dmgperturn` к иконке здоровья игрока; если иконку-источник найти нельзя, игра не подменяет источник иконкой атаки врага.

Батарея (`battleUse: "battery"`) является специальным результатом 5+ и не составляет обычные комбинации, даже если рядом есть другие батареи. Если игрок объединяет батарею с соседним предметом, активируются все предметы этого `type` на видимом поле, а сама батарея тоже исчезает. Эти активации применяют обычные `damage`, `heal` и `calm`, но не начисляют `aggression`. Если объединить батарею с батареей, активируются все предметы на видимом поле без начисления агрессии.

Предметы категории `item` могут усиливать матч-3 предметы через `modificate`. Например, если в инвентаре есть 3 `item_Shield`, а в его `modificate` указано `Shield.calm: 1` и `Shield_power.calm: 2`, то при активации `Shield` получит +3 к `calm`, а `Shield_power` получит +6 к `calm`. Модификаторы считаются в момент активации предмета и не изменяют сам каталог `items.jsonc`. После этих добавок текущая стадия врага может умножить итоговые `damage`, `heal`, `aggression` или `calm` через `stages[].itemStatModifiers`.

Предметы категории `item` могут менять случайную генерацию через `transform_chance`, `transform_from_itemId` и `transform_to_itemId`. Шанс считается как `transform_chance * количество предмета в инвентаре` и ограничивается 100%. Например, если у игрока 3 `item_Shield_power`, а у предмета указано `transform_chance: 1`, `transform_from_itemId: "Shield"`, `transform_to_itemId: "Shield_power"`, то каждая новая генерация `Shield` с шансом 3% станет `Shield_power`. Это применяется только к новым предметам при стартовой генерации, доборе и пополнении резервного поля; уже лежащие на поле предметы не превращаются задним числом. Если на тот же новый предмет действует `convert` текущей стадии врага, сначала проверяется шанс стадии. Только предмет, который не был превращен врагом, может пройти через трансформацию инвентаря игрока.

Предметы категории `item` могут пассивно менять боевые параметры игрока через `max_hp_modif` и `heal_hp_modif`. `max_hp_modif` увеличивает эффективный максимум здоровья в бою, а `heal_hp_modif` увеличивает количество HP, которое возвращается при заполнении шкалы лечения. Эти значения считаются от текущего количества предметов в инвентаре и не переписывают базовое состояние игрока в JSON.

Боевой модуль обрабатывает совпадения в строгом порядке: найти все совпадения, объединить активированные клетки, применить эффекты, определить бонусы, удалить активированные клетки, поставить бонусы, выполнить падение, добрать новые предметы и только потом обрабатывать каскады.

Стартовая генерация match-3 поля должна избегать начальных совпадений. Обычный обмен соседних предметов поддерживает два способа ввода: два коротких клика по исходной и соседней клетке либо свайп/перетаскивание исходного предмета в сторону соседнего. Доступные способы задаёт `settings.controlScheme`; по умолчанию `swipe-and-click` включает оба. Свайп определяется только после порога движения, всегда выбирает одну соседнюю клетку по преобладающей оси и выполняет тот же action flow, что два клика; поэтому стены, лозы, отмена обмена без совпадения, батареи и tutorial-ограничения работают одинаково. В режиме `click` полноценное перетаскивание подавляется, а в режиме `swipe` короткий клик по обычному полю не выбирает клетку. Во время активного черепа, золота или перчатки свайп не перехватывается: эти предметы сохраняют собственный кликовый сценарий при любой настройке и не расходуются случайно. Если игрок бездействует в бою, battle-view через `findBattleAvailableMove()` ищет возможный ход снизу вверх и слегка трясет именно тот предмет, который нужно передвинуть в комбинацию. Порядок поиска подсказки настраивается в `data/settings/battle-ui.jsonc` через `availableMoveSearch.typeGroups`: каждый вложенный список содержит типы, которые ищутся одновременно, а следующий список используется только если в предыдущем ничего не найдено. Значение `"*"` означает любые остальные типы. По умолчанию сначала ищутся `granate`, `Knife`, `bullet`, затем `Bandage`, `Shield`, затем все остальное. Тайминг подсказки (`idleHintDelayMs`, `idleHintShakeMs`) и время показа окна без ходов (`noMovesMessageMs`) задаются в `data/settings/battle-ui.jsonc`. Если ходов нет, бой только показывает предупреждение через `battle.noMoves.title` и `battle.noMoves.body`; автоматического урона и автоматического перемешивания больше нет. Это предупреждение не переводит бой в `isResolving`, поэтому кнопка ручного перемешивания остается доступной даже пока окно показано.

Под описанием ульты/ярости врага есть кнопка ручного перемешивания поля. Она использует видимый shuffle текущей доски и наносит игроку урон в размере атаки врага. Перемешивание выбирает вариант, где предметы уходят дальше от старых мест и где соседние куски поля меньше сохраняют прежнюю форму; это нужно, чтобы shuffle выглядел как сильное перемешивание, а не как перенос строк или групп. Сначала рендерится новая матрица, затем её иконки идут из старых позиций по двухсегментной детерминированной cubic Bezier-траектории: они стартуют быстро, плавно замедляются к концу и завершают полёт с `translate(0, 0)` уже в реальной целевой клетке. Поэтому после анимации не требуется визуальная подмена позиции и не возникает скачок. Визуальные перемещения предметов считаются от фактических локальных позиций DOM-ячеек, поэтому при shuffle, swap и падении иконки должны оставаться привязанными к сетке даже при fullscreen/cover-масштабе боя. Длительность перемешивания задается через `animations.noMovesShuffleMs` (сейчас 500 мс). Подпись кнопки задается через `battle-ui.textKeys.shuffleBoard`, а сообщение после нажатия - через `battle-ui.textKeys.shuffleBoardDone`.

При старте боя battle-модуль делает runtime JSON-снимок состояния игрока в `battleState.initialPlayerState`. Это не физический файл в `data/player`, потому что браузерная игра не может сама записывать файлы проекта. Если игрок погибает и нажимает `Начать сначала`, текущий бой запускается заново, а `playerState` восстанавливается из этого снимка: здоровье, лечение, инвентарь и временные боевые траты возвращаются к состоянию на входе в бой. Перезапуск создает новый attempt-token внутри того же боевого окна: старые async-анимации, каскады и таймерные цепочки от проигранной попытки не могут менять новое поле или блокировать таймер ярости. Постоянные кнопки боевого окна, включая ручное перемешивание, журнал, настройки и сдачу, при нажатии используют актуальные render targets новой попытки, а не отмененный token прошлой попытки.

Для стартового боевого поля нужно минимум 3 разных `type` среди обычных предметов `category: "match-3"`, которые могут случайно падать сверху. Валидатор перед START проверяет это по `data/settings/items.jsonc`, а `battle-engine` дополнительно проверяет фактические типы генерации с учетом `transform_chance` и текущего инвентаря игрока. Если обычных типов слишком мало или трансформации схлопнули генерацию до 1-2 типов, бой не пытается 50 раз собрать невозможное поле, а показывает понятную ошибку про слишком малое количество match-3 типов.

Размер match-3 поля задается в `data/settings/battle-ui.jsonc` через `board.width` и `board.height`. По умолчанию это 12 клеток по горизонтали и 9 по вертикали. Перед START значения должны быть целыми числами от 3 до 30; runtime все еще держит защитный fallback, но рабочий конфиг должен проходить явную проверку, чтобы ошибки размера не прятались.

Масштаб всего окна боя задается в `data/settings/battle-ui.jsonc` через `layout`. Бой верстается в виртуальном размере `layout.designWidthPx` x `layout.designHeightPx`, а затем целиком масштабируется в режиме cover: `.battle-scaffold-panel` должен покрывать весь viewport, а не оставлять поля вокруг окна боя. `layout.viewportPaddingPx` в рабочем конфиге поэтому равен `0`, а `layout.upscaleFactor` равен `1`; при несовпадении пропорций экрана с виртуальным размером бой может слегка обрезаться по одному краю, зато окно сражения остается полноэкранным. `layout.minScale` должен оставаться низким аварийным пределом: если поставить его слишком высоко, мобильные экраны снова не смогут уместить бой по высоте. В DOM для этого используется внешний `.battle-scaffold-frame`: он получает уже отмасштабированную ширину/высоту, а внутренняя `.battle-scaffold-panel` масштабируется через `transform`. Wrapper нужен, чтобы браузер центрировал видимый размер боя, а не невидимую исходную панель `1500x860`. Имя врага в верхней боевой шапке остается в одну строку и подбирает размер шрифта под доступную ширину своей плашки, чтобы текст был виден полностью и не растягивал окно врага вниз. Плашки мини-меню и сумки живут отдельными `fixed`-оверлеями, поэтому `battle-view` отдельно передает им текущий масштаб боя и производные размеры кнопок, отступов, слотов и чисел количества. Панель сумки дополнительно ограничивается правым краем текущего боевого wrapper'а, чтобы при сжатии окна она не вылезала за пределы боя, и поднимается вверх на четверть своей расчетной высоты, чтобы нижний край меньше выходил за видимую область. Кнопки-якоря сумки и мини-меню внутри overlay всегда строятся квадратными по максимальной стороне исходной кнопки, потому что исходные кнопки могут быть трансформированы CSS-сдвигами. Кнопки, которые вызывает мини-меню, также закреплены квадратным CSS-контрактом: `aspect-ratio`, запрет растяжения grid/flex и нулевые минимальные размеры. Иконки mini-menu в overlay центрируются абсолютно от середины кнопки, чтобы при сильном уменьшении не зависеть от baseline/line-height поведения обычного `img`. Это правило не стоит заменять отдельными `vh/vw`-подстройками внутри боя без сильной причины, иначе интерфейс снова начнет выглядеть по-разному на разных мониторах.

Визуальная сцена ярости настраивается в `data/settings/battle-ui.jsonc`: `icons.lightGold` задает золотой светлячок, `animations.rageWaveMs` - длительность белой волны по предметам, `rageProjectileCount`, `rageProjectileMs`, `rageProjectileArcHeightPx`, `rageProjectileSpreadPx` и `rageProjectileSizePx` - количество, длительность, дугу, разброс и размер золотых светлячков. Для ярости `kamikaze` параметр `animations.kamikazeBurstDistancePx` задает дальность разлета красных светлячков от здоровья врага перед самоуроном.

Стены боя задаются в enemy JSON через `wall`. Игра создает указанное количество стен на случайных ребрах между соседними клетками поля; стена не занимает клетку и не является предметом. Стена не генерируется на ребре, если хотя бы одна из двух соседних клеток закрыта коробкой, но лианы рядом со стенами разрешены. Визуально стена размером с боевую клетку, центрируется прямо между соседними клетками и рисуется поверх предметов. Обычный swap через стену запрещен, а поиск доступного хода не предлагает заблокированные обмены. При попытке обмена через стену трясется выбранный предмет и сама блокирующая стена. Иконки стен берутся из `data/settings/battle-ui.jsonc` (`icons.wall`, `icons.wall_1`, `icons.wall_2`), где `wall_1` и `wall_2` анимируются поочередно. Скорость чередования задается `animations.wallToggleMs` (время отображения каждого кадра). Стены генерируются один раз при старте боя и не пересоздаются при смене стадии врага или перемешивании поля. Спец-предметы вроде черепа и свободного обмена остаются отдельными механиками.

Коробки боя задаются в enemy JSON через `box`. Игра закрывает указанное количество случайных клеток поля отдельным визуальным слоем поверх предмета; иконка берется из `data/settings/battle-ui.jsonc` `icons.box`. Предмет под коробкой остается в матрице поля, но клетка считается заблокированной и неактивной: ее нельзя выбрать, поменять местами обычным или свободным swap, использовать как цель батареи, включать в обычные совпадения или подсказки доступного хода, учитывать для `dmgperturn`, превращать ультой врага или считать источником урона ульты. При падении и доборе коробка удерживает свой предмет на месте, поэтому элементы не проходят сквозь закрытую клетку. Коробки генерируются один раз при старте боя и не пересоздаются при смене стадии врага или перемешивании поля. Если пустые клетки находятся ниже коробки, новые предметы визуально выпадают из позиции коробки, а не с верхнего края поля.

Лианы боя задаются в enemy JSON через `vines`. Игра накрывает указанное количество случайных клеток отдельным визуальным слоем поверх квадрата и предмета; иконка берется из `data/settings/battle-ui.jsonc` `icons.vines`. Лианы не могут появиться на клетках с коробками и не блокируют падение: предметы спокойно падают под лиану и могут уходить из-под нее при свободном обмене. Клетка с лианой участвует в обычных совпадениях и активируется, если стала частью комбинации, но не предлагается как доступный ход для перемещения, не может быть целью батареи, не выбирается обычным swap и не перемещается ручным shuffle. Исключения по управлению: череп может активировать предмет под лианой, а перчатка может свободно поменять его с другой клеткой. Лианы генерируются один раз при старте боя и не пересоздаются при смене стадии врага или перемешивании поля.

Предметы на руках справа от боевого поля задаются в `data/settings/battle-ui.jsonc` через `handItemIds`. Это массив `itemId` из `data/settings/items.jsonc`, который рендерится в указанном порядке. Перед START валидатор проверяет, что список не пустой и что все эти предметы существуют в каталоге.

Каскады боя имеют защитный лимит `MAX_CASCADE_STEPS`, чтобы плохой конфиг или редкая случайность не загнали бой в бесконечную обработку. Если лимит достигнут, а на поле все еще есть совпадения, бой не скрывает ситуацию: в статус хода и боевой лог добавляется локализованное сообщение `battle.status.cascadeLimitReached`.

Шкала `heal` не является простым значением "не выше максимума": при переполнении она должна лечить игрока и оставлять остаток прогресса. Рабочий стартовый порог лечения сейчас `100`. Пример: было `80/100`, пришло `25`, лечение срабатывает, а на шкале остается `5/100`.

Когда шкала `heal` срабатывает, бой показывает восстановление как отдельное действие: красные светлячки летят от иконки лечения к иконке здоровья, а затем появляется число восстановленного HP. Эта обратная связь показывается даже если здоровье игрока уже было полным, потому что игрок все равно активировал лечение. Урон по врагу также должен показывать светлячки и отрицательное число даже в тот ход, когда враг переходит на следующую стадию.

Правило match-3: совмещать можно не только одинаковые `itemId`, но и разные предметы с одинаковым `type`.

Инвентарь:

- `itemId` - идентификатор предмета.
- `quantity` - количество.

Золото тоже считается предметом инвентаря. В данных для золота используется `itemId: "gold"`, а его название, описание и иконки задаются в `data/settings/items.jsonc`.

## Настройки

Настройки по умолчанию находятся в `data/settings/default-settings.json`.

Поля:

- `musicVolume` - громкость музыки от `0` до `1`.
- `soundVolume` - громкость звуков от `0` до `1`.
- `language` - текущий язык.
- `controlScheme` - схема обычного управления match-3 полем: `swipe`, `click` или `swipe-and-click`. По умолчанию включены оба способа. Настройка сохраняется в `localStorage` вместе с громкостью и языком и передаётся в каждый новый бой через `BattleRequest.settings`.
- `languages` - список доступных языков.
- `rewardAnimationMs.clearMs` - сколько миллисекунд reward-окно держит `eventImage` четким.
- `rewardAnimationMs.blurMs` - сколько миллисекунд длится размытие `eventImage`.
- `rewardAnimationMs.iconDelayMs` - через сколько миллисекунд после открытия окна начинают появляться иконки наград.
- `rewardAnimationMs.iconZoomMs` - сколько миллисекунд иконки наград увеличиваются из маленького размера в крупный.
- `rewardAnimationMs.iconStaggerMs` - задержка в миллисекундах между несколькими иконками.
- `battleTooltipMs` - Time (ms) a battle item tooltip stays visible.
- `battleMusicCrossfadeMs` - Time (ms) for smooth music crossfade map <-> battle.
- `inlineItemIconEm` - размер иконок, встроенных в текст локалей через `{item:itemId}`. Значение задается в `em`: `1` примерно равно размеру буквы, `2.1` примерно в два раза крупнее текста.
- `tooltipFontSizePx` - размер шрифта окон подсказок предметов в пикселях. В описаниях tooltip поддерживаются переносы строк через обычный JSON-символ `\n`.
- `tooltipMaxWidthPx` - максимальная ширина окна подсказки предмета в пикселях.
- `audio.click` - звук клика.
- `audio.mainMenuMusic` - музыка главного меню.
- `audio.mapMusic` - музыка карты.

Если добавить новый язык:

1. Создать файл `data/locales/<language>.json`.
2. Добавить `<language>` в `settings.languages`.
3. В каждой локали добавить ключ `language.<language>` с названием языка.

Названия языков должны быть одинаково узнаваемыми во всех локалях, например `language.en: "English"` и `language.ru: "Русский"`.

Перед стартом игра проверяет все локали из `settings.languages`, а не только текущий язык. Если в одной из активных локалей не хватает ключа интерфейса, кампании, карты, названия предмета или описания предмета, игра не начнет забег и покажет список недостающих ключей.

`rewardAnimationMs` читается из `current-settings.json` при загрузке страницы и не перекрывается старыми настройками из `localStorage`. После изменения этих чисел нужно перезагрузить страницу, чтобы runtime заново прочитал JSON.

В `data/settings/battle-ui.jsonc` для исхода боя используются ключи `battle.outcome.victory`, `battle.outcome.defeat`, `battle.outcome.restart`, `battle.outcome.restartPending`. Там же в `textKeys` лежат ключи `battle.status.*` для строки статуса и боевого лога, включая выбор клетки, отмену обмена и итог обработанного хода. Победная полоса показывается в течение `animations.outcomeBannerMs`, после чего бой возвращает карте `victory` и награду врага. При поражении показывается такая же полоса с кнопками `ui.surrender` и `battle.outcome.restart`; кнопка повтора пересобирает поле, восстанавливает состояние игрока на вход боя и заново запускает таймеры боя.

## Локализация

Локали устроены максимально просто: один JSON-объект, где ключ ведет напрямую к тексту.

Пример:

```json
{
  "ui.yes": "Да",
  "ui.no": "Нет",
  "trader.text_01": "Текст торговца"
}
```

В данных карты и кампании вместо текста используются ключи локали: `nameTextKey`, `dialogTextKey`, `textKey`, `titleTextKey`, `messageTextKey`.

## Главное меню и интерфейс карты

Главное меню:

- фон `data/Assets/backgrounds/main_menu.png`;
- номер текущей сборки из корневого `version.json` в правом верхнем углу;
- кнопка старт;
- кнопка настройки;
- музыка главного меню.

Фон главного меню всегда покрывает весь viewport. Текстовый блок меню с названием и кнопками живет в отдельной виртуальной панели и использует тот же текущий UI scale, что и map UI: на маленьких окнах сжимается, на больших увеличивается мягко. Это масштабирует только элементы меню, а не фон.

Настройки доступны и из главного меню, и поверх карты. На карте верхняя панель содержит:

- `Сдаться` - открывает подтверждение выхода в главное меню;
- `Настройки` - открывает настройки поверх карты;
- `Лог` - открывает лог событий.

При сдаче показывается локализованный диалог:

- вопрос `surrender.question`;
- подтверждение `surrender.confirm`;
- отмена `surrender.cancel`.

HUD карты находится слева сверху и отображается вертикальной колонкой. Значение в каждой строке выровнено слева и рассчитано минимум на 8 символов, чтобы здоровье вида `100/100` не обрезалось. Базовые показатели задаются в `data/settings/items.jsonc` через `showInHudAlways` и `hudOrder`:

- здоровье `current/max`;
- золото;
- черепа;
- часы;
- перчатка/обмен.

Остальные предметы из инвентаря добавляются ниже базовых показателей только если они есть и их `quantity` больше `0`. Значения берутся из текущего состояния игрока в памяти, а иконки, названия и подсказки - из каталога предметов.

Лог событий открывается кнопкой `Лог` и хранит историю всей текущей браузерной сессии. Новый забег не очищает старые записи: вместо этого добавляется выделенный заголовок `Забег N`, после чего идут строки проверки данных, генерации карты, выбора точек, открытия магазина/лекаря, покупок, лечения и переходов между картами.

## Технические заметки

JSONC-файлы поддерживают комментарии строками `//`. Обычные `.json` файлы комментарии не поддерживают.

Если `.json` или `.jsonc` не парсится, окно ошибки показывает путь к файлу, строку и колонку, исходное сообщение парсера, короткую подсказку и фрагмент строки с отметкой `^`. Для JSONC комментарии отдельными строками сохраняют нумерацию строк при проверке; inline-комментарии после значения не поддерживаются и будут показаны как ошибка синтаксиса.

Русскоязычные Markdown/JSONC-комментарии должны оставаться в UTF-8. После правок документации можно запустить `node scripts/check-encoding.mjs`: скрипт ищет характерные признаки mojibake, включая короткие осколки вроде сломанного middle-dot, и падает, если текст снова был сохранен в неправильной кодировке.

Данные JSON/JSONC загружаются с параметром `?v=<version>` из `version.json`. Сам `version.json` перечитывается с `cache: "no-store"`, а остальные файлы не сбрасывают кэш каждый запуск. Это важно для packaged/Steam-сборки: один и тот же билд не должен заново тянуть все ресурсы только потому, что игрок запустил игру еще раз.

Если картинка персонажа не загрузилась, `src/map-module.js` пишет в консоль ошибку вида:

```text
Failed to load character image: исходный_путь -> итоговый_путь
```

Проект сейчас запускается как статический сайт, например через локальный сервер на `http://127.0.0.1:5173/index.html`.

Runtime state карты и боя не публикуется в `window.context` или `window.contex`. Для ручной отладки есть явный dev-хук `window.__wildwestDebug`, который включается только через `?debug=1`, `?wildwestDebug=1` или `localStorage.wildwestDebug = "1"`. При включенном debug после старта карты доступен `window.__wildwestDebug.map.state`, а после входа в бой - например, `window.__wildwestDebug.map.lastBattleRequest` и `window.__wildwestDebug.battle.context`. Без debug-флага helper удаляет старые глобальные aliases и не держит runtime state в `window`.

Автоматический browser-smoke запускается командой:

```bash
node scripts/browser-smoke.mjs
```

В составе полного локального чека его можно запускать так:

```bash
node scripts/check-project.mjs --with-smoke
```

Скрипт сам поднимает временный static server, запускает системный Chrome/Edge через DevTools Protocol и открывает игру с фиксированным seed и `debug=1`. Основной полный режим - `--start=smoke-test`: smoke вводит `iddqd`, ждет скрытую кнопку, запускает `SmokeTest`, затем проходит линейный маршрут `dialog -> skip -> shop -> heal -> reward -> battle -> boss`. В магазине он выбирает все доступные товары, подтверждает покупку и сверяет золото/инвентарь/HUD; в лечении сверяет HP и цену; в reward и battle-наградах сверяет начисление золота/опыта/здоровья/предметов и закрывает возможные окна level-up; в боях использует чит `win` и проверяет `BattleRequest`, battle seed, `enemyConfigUrl`, доску `9x8`, 72 клетки, 72 иконки и стартовый `battleTrace`.

Старые быстрые режимы остаются доступными вручную. По умолчанию используется `--start=battle`, то есть бой с `easy_Foxy`; при `--start=dialog` smoke выбирает ответ `battle_test` и проверяет бой с `test`; при `--start=shop`, `--start=heal` и `--start=reward` smoke выбирает ответ Свити, выполняет действие события и сверяет обновление состояния/HUD; при `--start=level-up` активирует test-cheat `lvl`, выбирает level-up награду и проверяет ее начисление. Во всех режимах проверяются отсутствие `window.context/window.contex`, отсутствие console/page/network ошибок и отсутствие unversioned запросов к `src/` и `data/`. Скриншот финального состояния сохраняется в `artifacts/browser-smoke-*.png`.

Полезные опции:

- `--headed` - показать окно браузера.
- `--keep-open` - оставить окно и сервер открытыми после проверки.
- `--chrome=<path>` - указать путь к Chrome/Edge, если автоопределение не сработало.
- `--seed=<seed>` - заменить стандартный smoke seed `A7K9M2QX4T8ZB3NC`.
- `--start=smoke-test|battle|dialog|shop|heal|reward|level-up` - выбрать режим smoke; `smoke-test` проходит служебную QA-карту целиком, `battle` проверяет Фокси, `dialog` проверяет диалог Свити и тестовый бой, `shop`, `heal` и `reward` проверяют диалог Свити и полный цикл соответствующего события, `level-up` проверяет читовый level-up и выбор награды.

## Контракт боевого модуля

Карта и бой держатся как два отдельных модуля. Карта запускает бой, передавая ему данные события, состояние игрока, каталог предметов, локаль, настройки и seed. Бой не должен напрямую открывать точки карты, менять `generatedMap`, двигать камеру карты или сам завершать кампанию.

Текущий контракт находится в `src/battle/battle-contract.js`.

Структура боевого модуля:

- `src/battle/battle-contract.js` - версия контракта, типы входа/выхода и общий helper результата.
- `src/battle/battle-module.js` - публичный вход `startBattle(request, options)`. Только собирает data/engine/view вместе и возвращает `BattleResult`.
- `src/battle/battle-engine.js` - чистая логика и состояние боя без DOM и без доступа к карте.
- `src/battle/battle-trace.js` - JSON-трасса попытки боя: стартовое состояние, действия игрока, финальный исход и выгрузка файла из окна боевого лога.
- `src/battle/battle-view.js` - фасад DOM-экрана боя: координирует вынесенные view/flow-слои, engine calls и возвращает результат боя карте.
- `src/battle/battle-data.js` - загрузка данных боя. Загрузчик принимает `loadJsonc` извне, чтобы не копировать общий механизм загрузки карты и не связывать модуль с `map-module.js`.
- `src/battle/battle-state.js`, `battle-board-actions.js`, `battle-resolution.js`, `battle-rage-flow.js`, `battle-shuffle-flow.js`, `battle-outcome-flow.js` и view-модули рядом с ними держат состояние попытки, действия поля, каскады, ярость, shuffle/no-moves, исходы и DOM-слои отдельно от фасада.

Карта передает в бой объект `BattleRequest`:

- `contractVersion` - версия контракта.
- `nodeId` - id точки карты, которая запустила бой.
- `nodeType` - `battle` или `boss`.
- В текущих данных boss-события строятся как battle-события карты; контракт всё еще разрешает `boss`, чтобы не ломать будущие данные, если боссу понадобится отдельный тип.
- `enemyId` - какой файл врага использовать: `data/enemy/<enemyId>.jsonc`.
- `enemyConfigUrl` - точный путь к загружаемому enemy JSONC. Карта передает его явно, чтобы battle seed и фактический конфиг врага совпадали в логе.
- `background` - конкретный задник боя из события карты.
- `seed`, `seedName` - производный battle seed и его человекочитаемое имя вида `battle:<mapId>:<nodeId>:data/enemy/<enemyId>.jsonc:<attempt>`.
- `playerState` - копия или управляемое состояние игрока на момент входа в бой.
- `itemCatalog` - каталог предметов из `data/settings/items.jsonc`.
- `locale`, `language`, `settings` - данные для текстов и настроек боевого экрана.
- `cheats` - необязательный runtime-объект активных читов. Карта передает сюда только battle-scoped команды из `data/player/cheats.json`; боевой экран не читает player JSON напрямую.
- `tutorial` - необязательный сценарий управляемого боя из battle-варианта карты. Обычные battle-события передают `null` и не получают блокировок поля или учебного оверлея.

Бой возвращает карте `BattleResult`:

- `outcome` - `victory`, `defeat`, `escaped` или `cancelled`.
- `nodeId`, `nodeType` - какая точка была обработана.
- `playerState` - новое состояние игрока после боя.
- `reward` - payload награды из enemy JSON. Карта показывает его через общее reward-окно и начисляет только после кнопки `ui.claimReward`.
- `rewards` - старое поле для совместимости контракта; текущая награда врага передается через `reward`.
- `logMessages` - сообщения, которые карта добавит в глобальный лог.
- `battleTrace` - машинно-читаемая трасса попытки боя: seed/config, стартовое состояние, действия игрока, финальный outcome и итоговое состояние. Поле нужно для debug/replay-анализа; карта не должна принимать игровые решения по trace.

Правило границы: карта владеет прогрессом маршрута, доступными точками, завершением карты и глобальным логом. Бой владеет только временным боевым состоянием: врагом, стадией, полем, выбранными клетками, каскадами и боевыми эффектами. Это нужно, чтобы боевой экран можно было тестировать отдельно в sandbox-режиме и не ломать стабильную карту.

Если язык меняется через настройки во время боя, карта уведомляет активный `battle-view` через callback `onLanguageChange`. Бой обновляет свой `request.locale`, кнопки, имя врага, подписи шкал, tooltip-предметы и заголовок боевого лога без выхода обратно на карту.

## Battle Cascade Visual Notes

In battle cascades, the square board backgrounds stay fixed and only item icons animate. The battle board uses one grid: `.battle-scaffold-cell` elements are the static clickable square backgrounds, and `.battle-cell-icon` elements are separate direct grid items placed in the same row and column above those squares. The square visual is drawn by `.battle-scaffold-cell::before`; the cell itself must stay transparent and without `backdrop-filter`, otherwise static icons can look like they are under the square. Do not nest icons back inside cells, because cell backgrounds can cover moving icons during swap. The fall speed is configured in `data/settings/battle-ui.jsonc` through `animations.boardDropMs`, measured as milliseconds per board row. `animations.boardMoveStepMs` remains a fallback for older configs. When a 4- or 5-match creates a powered item or generator, the battle view prefers the cell where the player moved the piece; if that cell is not part of the match, the engine falls back to the first cell in the matched group.

The visible battle board has an invisible reserve board above it. The reserve board has the same width and height, is stored as battle state data, and is not checked for match-3 combinations. When visible items disappear, remaining visible items and the lower reserve items fall down together into the visible board; then the reserve board is refilled with new random `category: "match-3"` items. This keeps the visual refill readable: new items are treated as coming from a pseudo-real upper board, not as instant random replacements inside visible empty cells.

For visual stacking, the first new item starts `animations.newItemSpawnOffsetPx` pixels above the visible board top edge. Each next new item in the same column starts one full board-cell height above the previous new item, plus `animations.newItemStackGapPx` from `data/settings/battle-ui.jsonc`. `newItemStackGapPx` is an extra gap, not the full distance between spawned items.

All icon moves in one gravity/refill wave use a synchronized finish time. The view computes the longest movement in the wave and applies that duration to every moving icon, so short new-item drops do not visually settle before longer old-item drops from the same cascade step.

Battle meters have change feedback for player HP, enemy HP, player heal and enemy aggression.
When a value changes after initial render, the corresponding icon shakes and scales, and a green `+N` or red `-N` float appears near the icon.
HP uses `animations.healthChangeMs`, `animations.healthChangeScale`, and `animations.healthChangeFloatMs` / `animations.healthChangeFloatRisePx`.
Heal uses `animations.healChangeMs`, `animations.healChangeScale`, and `animations.healChangeFloatMs` / `animations.healChangeFloatRisePx`.
Enemy aggression uses `animations.aggressionChangeMs`, `animations.aggressionChangeScale`, and `animations.aggressionChangeFloatMs` / `animations.aggressionChangeFloatRisePx`.
This is UI feedback only and must not change combat math.

## Battle Special Items

`item_skull` and `item_swap` are active-use battle items. Clicking one consumes one item and shows its small marker near the cursor. Clicking the same active item again cancels the mode and refunds the item. While one of these modes is active, the other special item and `item_time` are unavailable.

`item_skull` activates the clicked board cell and the eight neighboring cells. These cells apply their normal effects except `aggression`, then disappear and trigger gravity/refill/cascades. `item_swap` lets the player select any two board cells and swap them freely, even when the swap does not create a match. New cells produced by refill are filled from lower rows upward, so the visual refill reads from bottom to top.

`gold` is also an active-use battle tool. Clicking the gold slot takes gold "in hand" without spending it yet. While gold is active, hovering a board item shows its `goldprice`; if the item has no valid `goldprice`, the hover marker shows a red `X` and the item cannot be changed. Clicking a valid target spends that `goldprice`, replaces the item with a random item from `data/settings/items.jsonc` where `goldloot: 1`, then applies the same player inventory `transform_chance` rules used by ordinary battle drops. The final result must differ from the source `itemId`: using gold on `granate` cannot leave `granate` in the same cell, even if `granate` itself has `goldloot: 1`. This keeps upgrades such as normal knife -> powered knife consistent with the rest of battle generation without making gold feel like a paid no-op.

## Map UI Config

`data/settings/map-ui.jsonc` stores UI-only map presentation settings. It is loaded and validated before a run starts, together with campaign, item, enemy and battle UI data. The example with comments lives in `data/settings/map-ui.example.jsonc`.

Map overlay UI uses its own `layout` scale in `data/settings/map-ui.jsonc`. Overlay windows are laid out in a virtual `layout.designWidthPx` x `layout.designHeightPx` panel, then scaled to the viewport with the same soft-upscale formula as battle UI. Permanent map UI uses the same scale but stays anchored to viewport edges: HUD scales from the top-left corner, while top action buttons scale from the top-right corner. The map itself is not scaled by this system: `map-board`, nodes, paths, birds and drag-scroll keep their own real map geometry.

Current responsibilities:

- `layout` controls the adaptive scale of map UI. `viewportPaddingPx` keeps a safe edge gap, `allowUpscale` enables growth on large screens, `upscaleFactor` controls how much of the extra room is used, and `minScale` is the emergency lower bound for small windows.
- `orientation` управляет принудительным landscape-режимом на телефонах. Если `forceLandscapeOnPhones=true`, устройство похоже на телефон, есть touch/coarse pointer и viewport портретный, весь игровой слой визуально поворачивается на `rotateDegrees`, а карта и бой считают размеры окна как горизонтальные.
- `layout.*ScaleReductionDivisor` weakens shrink for specific UI groups. For example, when the common scale would shrink by 45%, divisor `1.5` shrinks that group by 30% instead. `mainMenuScaleMultiplier` additionally increases the main menu after this shrink calculation; the current value `1.2` makes the title and buttons 20% larger. `settingsMenuScaleReductionDivisor` and `settingsMenuScaleMultiplier` do the same for the settings window, but as an extra scale inside the already centered overlay, so reward, shop, heal and dialog windows keep their previous geometry. `settingsMenuFontScale` changes only fonts and inner controls of the settings window; the current value `1.5` makes settings text one and a half times larger without changing the configured panel width.
- `topButtons` controls map top-right button text keys, icon paths and icon sizes.
- `topButtons.*.iconSizePx` controls the icon size only; CSS keeps the button itself square and clamps the icon inside the slot.
- `nodes` controls visual map event icons: hover scale, idle glow image/size and hover glow image/size. The idle `activeLightIcon` is rendered under currently available map nodes while the pointer is not on them; on hover/focus it fades out and `hoverLightIcon` fades in. Locked/completed points do not show either glow, so they do not look interactive.
- `nodes.positionJitterPct.x/y` controls stable visual node scatter in map percent units. The scatter is deterministic from map id and node id, so nodes keep their positions between renders.
- `nodes.layoutPasses` controls optional barycentric layout passes after path generation. `0` keeps the generation order, which is the safest default for fewer crossings because path generation already avoids crossing edges where possible.
- `dialog` controls only shared map dialog backdrop presentation: background darkness and blur. The character image, size and position belong to each `dialog` event payload in the map JSON.
- `dialog.textLetterMs` задает скорость печати текста, а `dialog.answersFadeMs` - длительность появления вариантов ответа после полной фразы.
- `animation.enabled` enables or disables ambient map animation.
- `animation.counts` controls how many animations of each type may exist at once and how often new ones spawn.
- `animation.definitions` stores concrete animation variants. The first supported type is `bird`.

Bird animation rules:

- `headingDegrees` uses screen-game degrees: `0` is up, `90` is right.
- `frames` are the flap frames.
- `glideFrame` is shown while the bird glides.
- `glideDurationMs` chooses a random glide time before each flap cycle.
- `frameIntervalMs` controls flap frame switching speed.
- `spawnEdges` controls where birds appear: `left`, `right`, `top`, or `bottom`.
- `movementSpeedPxPerSecond` controls movement speed across the visible map screen.
- `rotateWithHeading` controls whether the source frames are rotated toward the flight heading. It is `false` for the current bird frames, so the asset keeps its authored orientation.
- `spriteAngleOffsetDegrees` adds a fixed visual rotation only when a specific sprite sheet needs correction.
- A first animation of each configured type spawns immediately after the map is rendered; later spawns follow `spawnIntervalMs`.

The map animation layer is attached to `mapBoard`, not to the viewport. Spawn uses the currently visible map rectangle, but after creation birds live in map coordinates and scroll together with the map. Birds are cleaned up only after leaving the full layout bounds of the map, not after leaving the current viewport. The layer clips overflow at map edges, so birds flying past the bottom or side of the map do not expand the scrollable map height. The layer is decorative only: it uses `pointer-events: none`, so it must not block clicks on map nodes, HUD, surrender, settings or log buttons.
